## 一、项目设计类

### Q1: 请介绍一下这个项目的核心创新点

**参考答案**：

这个项目的核心创新是 **工具自进化能力** —— Agent 不仅能使用预定义的工具，还能在运行时自动创建新工具。

**三大创新点**：

1. **工具动态生成**
   - 当 Agent 发现没有合适的工具时，会调用 `create_tool` 元工具
   - LLM 生成工具代码、JSON Schema、测试用例
   - 通过安全校验和测试后自动注册

2. **三层安全防护**
   - AST 静态分析：在执行前检测危险代码
   - 进程级沙箱：subprocess 隔离 + 环境变量清洗
   - 测试驱动验证：工具必须通过测试才能注册

3. **自修复机制**
   - 生成的代码有问题时，自动构建修复提示
   - 让 LLM 迭代修正，最多 3 轮
   - 模拟人类 "编码-测试-debug" 循环

**灵感来源**：
- LATM 论文：LLM as Tool Makers
- Voyager：Minecraft AI 的技能库设计

### Q4: 为什么把"元工具"(create_tool, list_all_tools)放在 Agent Loop 而不是 Registry？

**参考答案**：

**设计考量**：

1. **职责分离**：
   - Registry 负责"管理已有工具"
   - Agent Loop 负责"决策是否创造工具"
   - 创造工具是一个复杂流程，不适合放在简单的注册表中

2. **避免循环依赖**：
   - 如果 `create_tool` 在 Registry 中
   - 而 `create_tool` 需要调用 `Tool Forge`
   - 而 `Tool Forge` 需要把结果注册回 Registry
   - 会造成复杂的双向依赖

3. **元工具的特殊性**：
   - `create_tool` 不是普通工具，它是"创造工具的工具"
   - 执行流程完全不同：普通工具直接执行，元工具触发整个 Forge 流程

**代码体现**：

```python
def _handle_tool_call(self, tc: ToolCall) -> str:
    # 元工具：特殊处理
    if tc.name == "create_tool":
        return self._forge_tool(...)  # Agent Loop 直接处理
    elif tc.name == "list_all_tools":
        return self.registry.list_tools()
    # 普通工具：委托给 Registry
    else:
        return self.registry.dispatch(tc.name, tc.arguments)
```

### Q5: 项目的模块划分遵循了哪些设计原则？

**参考答案**：

**1. 单一职责原则 (SRP)**

| 模块 | 单一职责 |
|------|----------|
| `agent_loop.py` | 只负责 ReAct 循环决策 |
| `tool_maker.py` | 只负责解析 LLM 输出 |
| `security.py` | 只负责 AST 安全检查 |
| `executor.py` | 只负责沙箱执行 |

**2. 依赖倒置原则 (DIP)**

- `AgentLoop` 依赖抽象的 `ToolRegistry` 接口
- 可以轻松替换为内存版、数据库版

**3. 开闭原则 (OCP)**

- 添加新的安全检查：只需扩展 `_SecurityVisitor`
- 添加新的内置工具：只需修改 `BUILTIN_HANDLERS`
- 无需修改核心代码

**4. 组合优于继承**

- `AgentLoop` 通过组合持有 `LLMClient`, `Registry`, `Context`
- 而不是通过继承获得这些能力

**5. Composition Root 模式**

- `main.py` 是组合根，负责组装所有依赖
- 各模块不自己创建依赖，而是接收注入

## 二、代码实现类

### Q6: 请详细解释 AST 安全校验的实现原理

**参考答案**：

**核心思路**：在代码执行前，用 Python AST 模块解析代码，遍历语法树查找危险模式。

**实现代码**：

```python
import ast

class _SecurityVisitor(ast.NodeVisitor):
    def __init__(self):
        self.violations = []
    
    def visit_Import(self, node):
        """检查 import xxx"""
        for alias in node.names:
            module = alias.name.split(".")[0]
            if module in BLOCKED_MODULES:
                self.violations.append(f"Blocked import: {module}")
        self.generic_visit(node)
    
    def visit_ImportFrom(self, node):
        """检查 from xxx import yyy"""
        if node.module:
            top = node.module.split(".")[0]
            if top in BLOCKED_MODULES:
                self.violations.append(f"Blocked: from {node.module}")
        self.generic_visit(node)
    
    def visit_Call(self, node):
        """检查函数调用 eval(), exec() 等"""
        name = self._get_call_name(node)
        if name in BLOCKED_BUILTINS:
            self.violations.append(f"Blocked builtin: {name}()")
        self.generic_visit(node)

def validate_code(source: str) -> ValidationResult:
    try:
        tree = ast.parse(source)
    except SyntaxError as e:
        return ValidationResult(safe=False, violations=[f"SyntaxError: {e}"])
    
    visitor = _SecurityVisitor()
    visitor.visit(tree)
    return ValidationResult(safe=len(visitor.violations) == 0, violations=visitor.violations)
```

**黑名单配置**：

```python
BLOCKED_MODULES = {"os", "subprocess", "shutil", "sys", "socket", ...}
BLOCKED_BUILTINS = {"eval", "exec", "__import__", "compile"}
```

**为什么用 AST 而不是正则？**

1. 正则无法理解代码结构，容易误判
2. AST 能准确识别 `import os` vs `"import os"` (字符串)
3. AST 能处理复杂情况如 `from os import system`

### Q7: 沙箱执行器的进程隔离是如何实现的？

**参考答案**：

**核心实现**：

```python
def execute_code(code: str, timeout: int = 30) -> ExecutionResult:
    # 1. 确保沙箱目录
    sandbox_dir = Path("/tmp/self-evolving-sandbox")
    sandbox_dir.mkdir(parents=True, exist_ok=True)
    
    # 2. 写入临时文件（避免 shell 转义问题）
    code_file = sandbox_dir / "_exec_code.py"
    code_file.write_text(wrapped_code)
    
    # 3. subprocess 执行
    proc = subprocess.run(
        [sys.executable, str(code_file)],  # 用相同 Python 解释器
        capture_output=True,
        text=True,
        timeout=timeout,
        cwd=str(sandbox_dir),  # 工作目录限制
        env=_safe_env(),       # 清洗后的环境变量
    )
    
    # 4. 返回结果
    return ExecutionResult(
        success=proc.returncode == 0,
        stdout=proc.stdout[:MAX_OUTPUT],  # 截断
        stderr=proc.stderr[:MAX_OUTPUT],
    )
```

**环境变量清洗**：

```python
def _safe_env() -> dict:
    """只保留必要的环境变量"""
    env = {}
    # 只保留这些
    for key in ("PATH", "HOME", "TMPDIR", "LANG", "LC_ALL"):
        if val := os.environ.get(key):
            env[key] = val
    # 防止导入项目代码
    env["PYTHONPATH"] = ""
    env["PYTHONDONTWRITEBYTECODE"] = "1"
    return env
```

**关键安全点**：

| 机制 | 作用 |
|------|------|
| `subprocess.run` | 独立进程，崩溃不影响主进程 |
| `cwd=sandbox_dir` | 限制文件访问范围 |
| `env=_safe_env()` | 防止 API Key 泄露 |
| `timeout=30` | 防止死循环/资源耗尽 |
| 输出截断 | 防止内存爆炸 |

### Q8: Tool Maker 如何解析 LLM 生成的代码？如果格式不对怎么办？

**参考答案**：

**期望格式**：

```markdown
```python
def tool_name(param: str) -> str:
    """Description"""
    return result
```

```json
{"name": "tool_name", "description": "...", "parameters": {...}}
```

```test
{"param": "value1"}
{"param": "value2"}
```
```

**解析逻辑**：

```python
def parse_forge_response(llm_output: str) -> ForgeResult:
    # 1. 提取 Python 代码
    code_match = re.search(r"```python\s*\n(.*?)```", llm_output, re.DOTALL)
    if not code_match:
        return ForgeResult(success=False, error="No Python code block found")
    code = code_match.group(1).strip()
    
    # 2. 提取函数名
    func_match = re.search(r"def\s+(\w+)\s*\(", code)
    if not func_match:
        return ForgeResult(success=False, error="No function definition found")
    func_name = func_match.group(1)
    
    # 3. 提取 JSON Schema（可选）
    schema_match = re.search(r"```json\s*\n(.*?)```", llm_output, re.DOTALL)
    if schema_match:
        schema = json.loads(schema_match.group(1))
    else:
        # 自动从函数签名推断 Schema
        schema = _auto_schema(func_name, code)
    
    # 4. 提取测试用例（可选）
    test_cases = []
    test_match = re.search(r"```test\s*\n(.*?)```", llm_output, re.DOTALL)
    if test_match:
        for line in test_match.group(1).split("\n"):
            if line.strip():
                test_cases.append(json.loads(line))
    
    return ForgeResult(success=True, record=ToolRecord(...))
```

**容错机制**：

1. **Schema 缺失**：自动从 AST 推断参数类型
2. **测试缺失**：允许空测试（生产环境应该要求至少 1 个）
3. **格式错误**：返回错误，触发 LLM 自修复

### Q9: 工具注册表如何实现"热加载"？

**参考答案**：

**热加载的含义**：新注册的工具立即可用，无需重启 Agent。

**实现方式**：

```python
class ToolRegistry:
    def __init__(self, store: ToolStore):
        self._handlers: Dict[str, Callable] = dict(BUILTIN_HANDLERS)
        self._schemas: Dict[str, dict] = {...}
        self._agent_tools: Dict[str, ToolRecord] = {}
        self._load_persisted()  # 启动时加载
    
    def register(self, record: ToolRecord, persist: bool = True):
        """注册新工具 —— 立即生效"""
        # 1. 创建沙箱执行的 handler
        func_code = record.code
        func_name = record.name
        
        def handler(args, _code=func_code, _name=func_name):
            result = execute_function(_code, _name, args)
            return result.return_value or result.stdout
        
        # 2. 更新内存中的映射
        self._handlers[record.name] = handler
        self._schemas[record.name] = {...}
        self._agent_tools[record.name] = record
        
        # 3. 持久化到磁盘
        if persist:
            self.store.save(record)
        
        # 无需重启！下次 LLM 获取 tools 时会包含新工具
```

**关键设计**：

1. **闭包捕获**：`handler` 闭包捕获 `_code` 和 `_name`，确保每个工具有自己的代码
2. **立即更新**：`self._handlers[name] = handler` 直接修改字典，立即生效
3. **持久化分离**：内存操作和磁盘操作分离，持久化不影响热加载

### Q10: 自修复机制是如何工作的？最多修复几轮？为什么？

**参考答案**：

**工作流程**：

```
[Round 1] LLM 生成代码
    │
    ├── AST 检查失败 ──► 构建修复提示 ──► [Round 2]
    │
    ├── 沙箱测试失败 ──► 构建修复提示 ──► [Round 2]
    │
    └── 全部通过 ──► 注册成功
    
[Round 2] LLM 修复代码
    │
    ├── 仍失败 ──► 构建修复提示 ──► [Round 3]
    │
    └── 全部通过 ──► 注册成功
    
[Round 3] LLM 再次修复
    │
    ├── 仍失败 ──► 返回失败（放弃）
    │
    └── 全部通过 ──► 注册成功
```

**修复提示构建**：

```python
def build_fix_prompt(code: str, errors: List[str], attempt: int) -> str:
    return f"""
The tool code you generated has errors. Please fix them.

## Current Code (Attempt {attempt}/{MAX_FIX_ATTEMPTS})
```python
{code}
```

## Errors
{chr(10).join(errors)}

## Instructions
1. Fix the issues described above
2. Return the COMPLETE corrected function
3. Use the same output format
"""
```

**为什么是 3 轮？**

1. **经验数据**：大多数问题 1-2 轮就能修复
2. **成本控制**：每轮都是一次 LLM 调用，3 轮是平衡点
3. **避免死循环**：有些问题 LLM 确实无法修复（如不支持的库）
4. **用户体验**：3 轮已经等待较长时间，再多用户会失去耐心

**配置化**：

```python
# config.py
MAX_FIX_ATTEMPTS = 3  # 可根据需要调整
```

### Q11: 你的沙箱能防住哪些攻击？举例说明

**参考答案**：

| 攻击类型 | 攻击代码示例 | 防护层 |
|----------|--------------|--------|
| 命令注入 | `os.system("rm -rf /")` | AST 拦截 `os` |
| 代码注入 | `eval(user_input)` | AST 拦截 `eval` |
| 信息泄露 | `print(os.environ["OPENAI_API_KEY"])` | 环境变量清洗 |
| 文件窃取 | `open("/etc/passwd").read()` | AST 拦截绝对路径 |
| 资源耗尽 | `while True: pass` | timeout 超时 |
| 内存炸弹 | `"x" * (10**10)` | 输出截断 |
| 进程逃逸 | `subprocess.run(["ls"])` | AST 拦截 `subprocess` |

**具体防护代码**：

```python
# 命令注入防护
def visit_Import(self, node):
    if "os" in [a.name for a in node.names]:
        self.violations.append("Blocked import: os")

# 信息泄露防护
def _safe_env():
    # 只保留安全的环境变量
    return {k: os.environ[k] for k in ("PATH", "HOME", "TMPDIR") if k in os.environ}

# 文件访问防护
def visit_Call(self, node):
    if self._get_call_name(node) == "open":
        path = node.args[0].value
        if path.startswith("/") and not path.startswith("/tmp"):
            self.violations.append("File access outside sandbox")
```

### Q12: 如果 LLM 生成 `import os as o; o.system("rm -rf /")` 能检测到吗？

**AST 检测原理**：

# 攻击代码
import os as o

Import(names=[alias(name='os', asname='o')])

# 检测逻辑

### Q14: 为什么先做 AST 检查再做沙箱执行？能不能只用沙箱？

**参考答案**：

**为什么要 AST 先行**：

1. **快速失败**：AST 检查是纯计算，毫秒级完成
2. **无副作用**：不执行代码，不消耗沙箱资源
3. **明确报错**：能精确指出 "第 5 行使用了 os.system"

**如果只用沙箱**：

```
危险代码 → 沙箱执行 → 发现错误（但可能已造成影响）
```

问题：
- 某些危险操作可能不报错但有副作用
- 沙箱隔离不是 100% 可靠
- 启动进程的开销白白浪费

**正确的流水线**：

```
代码 → AST检查(快,无副作用) → 沙箱执行(慢,有隔离)
         │                          │
         ▼                          ▼
      拒绝危险代码              执行安全代码
```

**类比**：

## 四、Agent 原理类

### Q15: 什么是 ReAct 循环？你的项目是如何实现的？

**参考答案**：

**ReAct 原理**：Reasoning + Acting

```
Think → Act → Observe → Think → Act → Observe → ... → Final Answer
```

**核心思想**：
- **Think**：LLM 思考下一步该做什么
- **Act**：调用工具执行动作
- **Observe**：将工具结果反馈给 LLM
- 循环直到 LLM 给出最终答案

**本项目实现**：

```python
def run(self, user_input: str) -> str:
    self.context.add_user(user_input)
    
    for turn in range(MAX_TURNS):
        # Think: 调用 LLM
        action = self.llm.chat(self.context.messages, tools=all_tools)
        
        # 检查是否结束
        if action.is_final:  # LLM 没有调用工具，直接给出答案
            return action.text
        
        # Act: 执行工具
        for tc in action.tool_calls:
            result = self._handle_tool_call(tc)
            # Observe: 将结果加入上下文
            self.context.add_tool_result(tc.id, result)
        
        # 压缩上下文（如果太长）
        if self.context.check_compress():
            self.context.compress()
    
    return "(max turns reached)"
```

**与传统 Agent 的区别**：

| 传统 Agent | 本项目 Agent |
|------------|--------------|
| 只能用现有工具 | 可以创建新工具 |
| 工具集固定 | 工具集动态增长 |

### Q17: 为什么需要上下文压缩？如何实现？

**参考答案**：

**为什么需要**：

1. LLM 有上下文窗口限制（如 GPT-4o 是 128K tokens）
2. 工具调用会累积大量信息
3. 超出限制会导致 API 报错

**压缩策略**：

```python
class ContextManager:
    def __init__(self, system_prompt: str, max_tokens: int = 100_000):
        self.messages = [{"role": "system", "content": system_prompt}]
        self.max_tokens = max_tokens
    
    def check_compress(self) -> bool:
        """检查是否需要压缩"""
        total_len = sum(len(m.get("content", "")) for m in self.messages)
        return total_len > self.max_tokens * 0.8  # 80% 阈值
    
    def compress(self):
        """压缩上下文"""
        # 方案 1: 截断旧消息
        while self._estimate_tokens() > self.max_tokens:
            # 保留 system prompt 和最近的消息
            if len(self.messages) > 2:
                del self.messages[1]
        
        # 方案 2: 摘要压缩（更高级）
        # summary = self.llm.summarize(self.messages[1:-5])
        # self.messages = [self.messages[0], {"role": "assistant", "content": summary}] + self.messages[-5:]
```

**本项目采用的策略**：

1. 简单截断：删除最早的消息
2. 保留 system prompt
3. 保留最近的几轮对话

### Q21: 如何处理 LLM API 的流式响应？

**参考答案**：

**流式响应的作用**：
- 边生成边返回，用户体验更好
- 长响应不会超时

**实现方式**：

```python
def stream_chat(self, messages, tools=None):
    stream = self.client.chat.completions.create(
        messages=messages,
        tools=tools,
        stream=True,  # 开启流式
    )
    
    full_content = ""
    tool_calls_buffer = {}
    
    for chunk in stream:
        delta = chunk.choices[0].delta
        
        # 文本内容
        if delta.content:
            full_content += delta.content
            yield {"type": "content", "content": delta.content}
        
        # 工具调用（流式需要累积）
        if delta.tool_calls:
            for tc_delta in delta.tool_calls:
                idx = tc_delta.index
                if idx not in tool_calls_buffer:
                    tool_calls_buffer[idx] = {
                        "id": "", "name": "", "arguments": ""
                    }
                if tc_delta.id:
                    tool_calls_buffer[idx]["id"] = tc_delta.id
                if tc_delta.function:
                    if tc_delta.function.name:
                        tool_calls_buffer[idx]["name"] = tc_delta.function.name
                    if tc_delta.function.arguments:
                        tool_calls_buffer[idx]["arguments"] += tc_delta.function.arguments
    
    # 最终结果
    if tool_calls_buffer:
        yield {"type": "tool_calls", "calls": list(tool_calls_buffer.values())}
    elif full_content:
        yield {"type": "final", "content": full_content}
```

**注意点**：
- 工具调用是分块返回的，需要累积
- `arguments` 是 JSON 字符串的片段，要拼接后再解析

class MetricsCollector:

   - 解决：增加更多测试用例、代码复杂度检查

3. **工具膨胀**：用户创建大量工具，占用存储
   - 解决：限制每用户工具数量、自动清理不用的工具
