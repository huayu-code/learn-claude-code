# 实习经历 — AI Agent 沙箱代码执行引擎 面试题

---

## 一、整体架构理解（了解性）

### Q1: 简单介绍一下你做的沙箱代码执行引擎，它在整个产品中扮演什么角色？

**答案**：
沙箱引擎是 AI Agent 的核心基础设施，负责在安全隔离的环境中执行 LLM 生成的代码。在整个产品链路中，用户发起对话 → LLM 决策需要执行代码 → 路由到沙箱引擎 → 创建/复用沙箱 → 执行代码并返回结果 → 结果回传给 LLM 继续推理。它相当于给 AI 提供了一个"可以动手做事"的安全工作台。

**建议**：回答时要从产品视角出发，先说清楚"为什么需要沙箱"（安全隔离执行不可信代码），再说你负责的范围。

---

### Q2: 你用的技术栈是 Go + tRPC-Go + AGS SDK，能分别说下它们在架构中的角色吗？

**答案**：
- **Go**：主语言，利用 goroutine/channel 天然适合高并发 I/O 密集型场景（沙箱管理涉及大量 RPC 调用和异步事件）
- **tRPC-Go**：腾讯内部 RPC 框架，负责服务间通信，包括沙箱状态的远程持久化、COS 服务调用等
- **AGS SDK**：Agent Gateway Service 的 SDK，封装了沙箱的创建、连接、执行、销毁等底层操作，我在其上层做生命周期管理和容错

**建议**：面试官可能不了解 tRPC-Go 和 AGS SDK，用"类比"解释——tRPC-Go 类似 gRPC，AGS SDK 类似沙箱的 Docker Client。

---

### Q3: 沙箱的完整生命周期是怎样的？从创建到销毁的流程？

**答案**：
```
路由（判断是否需要沙箱）
  → 创建（调 AGS SDK 创建沙箱实例）
  → 绑定（将沙箱与当前会话绑定，写入 Session State）
  → 复用（后续请求命中同一会话时，直接从 Session State 取沙箱实例）
  → 执行（通过 SDK 连接发送代码执行指令）
  → 销毁（30 分钟空闲自动回收 / 临时沙箱 5 分钟超时销毁）
```

**建议**：画图最清晰，面试时可以边说边画流程图。重点强调"复用"环节，因为这是你的核心优化点。

---

## 二、Session State 双层持久化（核心亮点 — 深度题）

### Q4: 为什么需要"双层持久化"？只用本地内存或只用远程 RPC 不行吗？

**答案**：
- **只用本地内存**：服务重启或扩缩容时沙箱状态丢失，用户下一次请求会找不到之前的沙箱，被迫重建
- **只用远程 RPC**：每次查询沙箱状态都要走网络调用，在高频调用场景下延迟不可接受（每次代码执行前都要查一次）
- **双层方案**：本地内存作为热缓存（O(1) 查询），远程 RPC 作为持久化兜底（容灾恢复）。写入时双写，读取时优先本地 → miss 则回源远程 → 回填本地

**建议**：这是经典的"缓存 + 持久化"架构，面试官很可能追问一致性问题。

---

### Q5: 双层之间的数据一致性怎么保证？如果本地和远程不一致怎么办？

**答案**：
- 写入采用**先远程后本地**策略：远程 RPC 写成功后才更新本地内存，远程写失败则不更新本地（保证远程是 source of truth）
- 读取时本地 miss 会回源远程并回填，天然保证最终一致
- 沙箱本身有 TTL（30 分钟 / 5 分钟），即使出现短暂不一致，过期后会自动清理
- **极端场景**：如果远程写成功但本地更新前进程 crash，下次请求会 miss 本地 → 回源远程 → 正常恢复

**建议**：强调"最终一致性"而非"强一致性"，因为沙箱场景不需要强一致，TTL 机制天然兜底。

---

### Q6: "沙箱创建频次降低 80%+"这个数据怎么来的？

**答案**：
通过 OpenTelemetry 埋点统计。对比优化前后一段时间内的沙箱创建 API 调用次数：
- 优化前：每次代码执行请求都触发沙箱创建
- 优化后：同一会话内的多次代码执行复用同一个沙箱，只有首次和沙箱过期后才创建
- 实际观测到平均一个会话有 5+ 次代码执行，所以创建频次至少降低 80%

**建议**：面试官追问数据来源是常见套路，一定要能说清楚怎么测的、对比基准是什么。如果没有精确数据，用逻辑推导也可以。

---

### Q7: Session State 的数据结构是怎样的？存了哪些字段？

**答案**：
核心字段包括：
- `sandbox_id`：沙箱实例唯一标识
- `session_id`：会话 ID（与用户对话绑定）
- `connection`：SDK 连接对象（本地内存独有，不可序列化）
- `status`：沙箱状态（creating / ready / executing / expired / error）
- `created_at`：创建时间（用于 TTL 计算）
- `last_active_at`：最后活跃时间（用于空闲超时判断）

远程持久化时 `connection` 不存储，恢复时需要重新建立连接。

**建议**：说清楚"哪些字段可序列化、哪些不可序列化"，体现你对持久化边界的理解。

---

## 三、三级连接降级策略（深度题）

### Q8: 三级降级策略具体是怎么实现的？每一级的触发条件是什么？

**答案**：
```
Level 1: SDK 正常连接
  → 直接通过 AGS SDK 建立 WebSocket/gRPC 连接到沙箱
  → 失败条件：连接超时、网络异常、SDK 内部错误

Level 2: 健康检查
  → 连接建立但不确定沙箱是否可用时，发送健康检查探针
  → 检查沙箱进程存活 + 文件系统可写 + 基本执行能力
  → 失败条件：健康检查超时或返回异常

Level 3: 自动重建
  → 健康检查也失败，说明沙箱实例已不可用
  → 销毁旧沙箱 → 清理 Session State → 创建新沙箱 → 重新绑定
  → 对用户透明，上层只感知到短暂延迟
```

**建议**：这个回答要体现"渐进式降级"的思想——先尝试最轻量的恢复（重连），不行再做更重的操作（重建）。

---

### Q9: "故障恢复时间 < 3s"怎么做到的？重建沙箱不需要很长时间吗？

**答案**：
- Level 1→2 的切换几乎是瞬时的（健康检查超时一般设 1-2s）
- Level 3 重建时，沙箱创建本身约 1-2s（AGS 平台预热池机制）
- 关键优化：**异步预创建**——在健康检查阶段就预先触发新沙箱创建，如果健康检查通过则丢弃预创建的沙箱，如果失败则直接使用预创建的沙箱，省去等待时间
- 总耗时：健康检查超时（~1s）+ 预创建已就绪（~0s）+ 状态切换（~0.5s）< 3s

**建议**：如果没有做预创建优化，可以诚实说"重建约 2-3s，在可接受范围内"。面试官更看重你的思考而非虚构优化。

---

### Q10: "失败不阻塞主流程"具体是什么意思？沙箱创建失败了用户还能正常用吗？

**答案**：
这里的"不阻塞"有两层含义：
1. **COS 初始化/模板初始化失败不阻塞**：沙箱创建的主流程是"创建实例 + 建立连接"，COS 挂载、模板文件初始化是附加步骤，失败后记录日志但不 abort 整个流程，沙箱仍可使用（只是没有预置文件）
2. **远程持久化失败不阻塞**：Session State 写入远程失败时，本地内存仍有状态，当前会话不受影响，降级为"重启后不可恢复"

**建议**：这体现了"优雅降级"的工程思想——区分核心路径和非核心路径，非核心失败不应该拖垮整个系统。

## 四、SSE 事件管道（深度题）

### Q11: 为什么用 SSE 而不是 WebSocket 来推送沙箱状态？

**答案**：
- **单向性**：沙箱状态推送是服务端→客户端的单向流，不需要客户端向服务端发消息，SSE 天然适合
- **HTTP 兼容性**：SSE 基于 HTTP，穿透代理/负载均衡更容易，不需要额外的 WebSocket 升级握手
- **自动重连**：浏览器原生 EventSource 有自动重连机制
- **简单性**：相比 WebSocket，SSE 的服务端实现更简单，不需要处理心跳、分帧等

**建议**：如果实际用的是 WebSocket 或其他协议，如实说明。关键是说清楚"为什么选这个"。

---

### Q12: 1024 缓冲区是什么意思？为什么是 1024？

**答案**：
指的是 Go channel 的缓冲区大小 `make(chan Event, 1024)`。沙箱执行过程中会产生大量事件（创建中、连接中、执行中、输出行、执行完成等），生产端（沙箱执行 goroutine）和消费端（SSE 推送 goroutine）速率不一致，需要缓冲区解耦。

1024 的选择：
- 太小（如 10）：沙箱执行产生事件快于 SSE 推送时会阻塞执行 goroutine
- 太大（如 100000）：浪费内存，每个会话一个 channel
- 1024：经验值，足以容纳一次完整执行的所有事件（通常几十到几百个），留有余量

**建议**：面试官可能追问"channel 满了怎么办"——可以回答用 `select + default` 非阻塞发送，满了就丢弃或降级。

---

### Q13: 10+ 种生命周期事件具体有哪些？

**答案**：
```
sandbox_creating    — 沙箱创建中
sandbox_ready       — 沙箱就绪
sandbox_connecting  — 正在建立连接
sandbox_connected   — 连接成功
code_executing      — 代码执行中
code_stdout         — 标准输出（可能多条）
code_stderr         — 标准错误输出
code_completed      — 执行完成
code_timeout        — 执行超时
code_error          — 执行错误
sandbox_destroying  — 沙箱销毁中
sandbox_destroyed   — 沙箱已销毁
file_uploaded       — 文件上传完成
cos_mounted         — COS 挂载完成
```

**建议**：不需要背全部，但至少能说出 7-8 种，体现你对整个生命周期的掌握。

## 五、COS 挂载与 OpenTelemetry（了解性）

### Q14: COS 可读写挂载是怎么实现的？为什么不直接在沙箱里用 SDK 读写 COS？

**答案**：
COS（对象存储）挂载是将远程存储映射为沙箱内的本地目录，让沙箱内的代码可以像操作本地文件一样读写 COS 上的文件。相比在沙箱内使用 SDK：
- **安全性**：不需要在沙箱内暴露 COS 的 SecretId/SecretKey
- **易用性**：LLM 生成的代码直接 `open("file.csv")` 即可，不需要学 COS SDK
- **通用性**：任何语言的代码都能用，不限于 Python/Node

**建议**：如果不了解挂载的底层实现（FUSE/CSI），可以说"这部分由 AGS 平台提供，我负责的是触发挂载和路径配置"。

---

### Q15: OpenTelemetry + Langfuse 分别解决什么问题？为什么两个都要？

**答案**：
- **OpenTelemetry**：通用的分布式追踪，覆盖 RPC 调用链路（沙箱创建→连接→执行→销毁），可以看到每个环节的耗时、错误率、调用关系。主要面向**后端工程师排查性能和故障**
- **Langfuse**：专门面向 LLM 应用的可观测性，追踪 Prompt→LLM 调用→Tool Calling→Agent 循环的全过程，可以看到 token 消耗、模型选择、Prompt 版本等。主要面向**AI 工程师优化模型效果**
- 两者互补：OpenTelemetry 管"系统层"，Langfuse 管"AI 层"

**建议**：这个问题体现你对可观测性的理解层次，不是简单堆工具，而是不同工具解决不同层面的问题。

## 六、设计决策 & 权衡（深度题）

### Q16: 沙箱 30 分钟自动回收、临时沙箱 5 分钟超时销毁，这两个时间怎么确定的？

**答案**：
- **30 分钟**：对应用户的一次"工作会话"。分析用户行为数据，大部分用户连续使用 AI Agent 的时长在 10-30 分钟之间。设 30 分钟是在"复用率"和"资源占用"之间的权衡——太短会导致频繁重建，太长会浪费服务器资源
- **5 分钟**：临时沙箱是为一次性任务创建的（非会话绑定），5 分钟足够完成一次代码执行，超时说明已废弃

**建议**：如果没有真实数据支撑，可以说"参考了业界实践（如 Google Colab 的 idle timeout）并结合资源成本估算"。

---

### Q17: 如果沙箱在执行代码的过程中网络断了，怎么处理？

**答案**：
分两种情况：
1. **客户端→服务端断连**：SSE 连接断开，但沙箱执行不受影响（执行在服务端）。客户端重连后通过 Session State 找到沙箱，查询执行结果
2. **服务端→沙箱断连**：SDK 连接中断。此时触发降级策略：
   - 尝试重连（Level 1）
   - 健康检查（Level 2）
   - 如果沙箱实例还在但连接断了，重建连接即可
   - 如果沙箱实例也挂了，走 Level 3 重建，但本次执行结果会丢失，需要告知用户重试

**建议**：这是分布式系统的经典问题，关键是区分"连接断 vs 实例挂"两种故障。

---

### Q18: 整个沙箱引擎的并发模型是怎样的？一个用户多次执行代码会并发还是排队？

**答案**：
- 单个沙箱内的代码执行是**串行的**（同一时刻只执行一段代码），因为共享文件系统和环境状态，并发执行会互相干扰
- 不同用户的不同沙箱之间是**并发的**，每个沙箱是独立的隔离环境
- 服务端本身用 Go goroutine 并发处理多个用户的请求，通过 Session State 路由到各自的沙箱

**建议**：面试官可能追问"如果用户快速提交多次执行怎么办"——回答用队列排队，前一个完成后再执行下一个。

---

### Q19: 你在做这个项目过程中遇到的最大技术挑战是什么？

**答案**：
（建议准备一个真实的踩坑经历，示例：）
最大挑战是**沙箱状态的一致性管理**。因为沙箱实例在远端（AGS 平台），状态在本地 + 远程两份，而沙箱可能被平台侧主动回收（资源不足时）。一开始没考虑"平台侧主动销毁"的场景，导致本地 Session State 还认为沙箱存活，实际已经被回收了，用户执行代码时报"沙箱不存在"的错。后来加了**心跳探测 + 状态同步**机制：定期检查沙箱是否存活，发现不存活时主动清理本地状态并走重建流程。

**建议**：一定要准备一个"真实踩坑 → 分析原因 → 解决方案"的故事，面试官最爱这种。

---

### Q20: 如果让你重新设计，你会做什么改进？

**答案**：
1. **沙箱预热池**：提前创建一批空闲沙箱，用户请求时直接分配，进一步降低首次创建延迟
2. **沙箱快照/恢复**：执行完代码后保存沙箱状态快照，下次恢复时不需要重新安装依赖
3. **更细粒度的资源限制**：CPU/内存/网络的 cgroup 级别隔离，目前主要依赖 AGS 平台的隔离能力
4. **多语言支持**：当前主要支持 Python，可以扩展到 Node.js、Go 等

**建议**：改进方向要**具体可落地**，不要说"用微服务拆分"这种空话。
