## 一、Agent 技术栈（核心 — 深度题）

### Q1: 解释一下 ReAct 循环？和普通的 LLM 调用有什么区别？

**答案**：
**普通 LLM 调用**：用户问 → LLM 答 → 结束（一问一答）

**ReAct（Reasoning + Acting）**：让 LLM 能够"思考 + 行动"的循环
```
Observe（观察）→ Think（思考）→ Act（行动）→ Reflect（反思）→ 循环...
```

具体流程：
1. **Observe**：接收用户输入或工具执行结果
2. **Think**：LLM 分析当前状态，决定下一步
3. **Act**：执行工具调用或生成回复
4. **Reflect**：检查结果，决定是否继续

区别：
| 维度 | 普通 LLM | ReAct Agent |
|------|---------|-------------|
| **交互** | 一问一答 | 多轮循环 |
| **能力** | 只能回答 | 能调用工具 |
| **复杂任务** | 一次完成 | 分步推理 |
| **错误处理** | 无 | 可自动重试 |

**建议**：ReAct 是 Agent 的核心架构，一定要能画出流程图。

### Q2: Function Calling / Tool Calling 的原理是什么？

**答案**：
**原理**：在 Prompt 中告诉 LLM 有哪些可用工具（JSON Schema 格式），LLM 根据用户请求决定是否调用工具。

**流程**：
```
1. System Prompt 中注入工具定义：
   "你有以下工具可用：
    - search_web: 搜索网页
    - run_code: 执行代码"

2. 用户："帮我搜一下今天的天气"

3. LLM 返回（结构化）：
   {"tool_calls": [{"name": "search_web", "args": {"query": "今天天气"}}]}

4. 系统执行工具，将结果追加到上下文

5. LLM 基于结果生成最终回复
```

**OpenAI 的 Function Calling 规范**：
```json
{
    "name": "search_web",
    "description": "搜索互联网获取信息",
    "parameters": {
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "搜索关键词"}
        },
        "required": ["query"]
    }
}
```

**建议**：能手写一个 Tool Schema，体现你对协议的理解。

### Q3: 什么是 Prompt Engineering？有哪些常用技巧？

**答案**：
**Prompt Engineering**：通过设计提示词来引导 LLM 产生期望的输出。

常用技巧：
| 技巧 | 说明 | 示例 |
|------|------|------|
| **角色设定** | 给 LLM 一个身份 | "你是一个资深Python工程师" |
| **Few-shot** | 给几个示例 | "输入：2+2，输出：4" |
| **Chain-of-Thought** | 要求分步思考 | "让我们一步一步思考" |
| **Output Format** | 指定输出格式 | "请用JSON格式返回" |
| **Negative Prompt** | 说明不要做什么 | "不要编造信息" |

**我在项目中的应用**：
- Self-Evolving Agent：要求 LLM "生成代码 + Schema + 测试用例"
- 客服系统：权限感知 Prompt "只读文档仅推荐分析类问题"

**建议**：Prompt Engineering 是 AI 应用的基本功，要有实际应用案例。

### Q4: 什么是 MCP（Model Context Protocol）？和 Tool Calling 有什么区别？

**答案**：
**MCP**：Anthropic 提出的模型上下文协议，用于标准化 LLM 与外部系统的交互。

**和 Tool Calling 的区别**：
| 维度 | Tool Calling | MCP |
|------|-------------|-----|
| **定位** | 工具调用机制 | 上下文管理协议 |
| **范围** | 调用函数/API | 资源、工具、Prompt 等全面管理 |
| **提出者** | OpenAI | Anthropic |
| **标准化** | 各厂商实现不同 | 统一标准 |

MCP 的核心概念：
- **Resources**：可供 LLM 访问的数据源（文件、数据库等）
- **Tools**：LLM 可调用的功能
- **Prompts**：预定义的提示词模板
- **Sampling**：LLM 自己发起请求的能力

**建议**：MCP 是新兴概念，了解即可，不需要深入。

### Q5: RAG 和 Fine-tuning 的区别是什么？什么场景用哪个？

**答案**：
| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| **原理** | 检索外部知识 + 生成 | 修改模型参数 |
| **知识更新** | 更新知识库即可 | 需要重新训练 |
| **成本** | 低（只需向量化） | 高（GPU 训练） |
| **可解释性** | 高（可追溯知识来源） | 低（黑盒） |
| **适用场景** | 知识密集型任务 | 风格/格式调整 |

选择指南：
- **用 RAG**：
  - 知识需要频繁更新（客服政策、商品信息）
  - 需要引用来源（法律、医疗）
  - 数据敏感不想发送给模型厂商

- **用 Fine-tuning**：
  - 需要特定的回答风格（品牌语气）
  - 特殊格式要求（SQL 生成、代码补全）
  - 小模型在特定任务上提升效果

**建议**：这是高频面试题，要能说清楚两者的 trade-off。

### Q6: 上下文窗口管理有哪些挑战？你是怎么解决的？

**答案**：
**挑战**：LLM 有 token 限制（如 GPT-4 128K），长对话会超出窗口。

**解决方案**：

1. **上下文压缩**（我在 Self-Evolving Agent 中用的）：
```python
if estimate_tokens() > 60000:
    # 保留最近 6 条消息
    # 旧消息让 LLM 生成摘要
    # 用摘要替换旧消息
```

2. **滑动窗口**：只保留最近 N 条消息

3. **记忆外挂**：
   - 短期记忆：上下文中的消息
   - 长期记忆：向量数据库存储历史，按需检索

4. **消息分级**：
   - 系统消息：始终保留
   - 工具结果：可压缩
   - 用户对话：按时间衰减

**建议**：上下文管理是长对话 Agent 的核心问题，要有实际解决方案。

### Q8: JVM 内存模型是什么？什么时候会发生 OOM？

**答案**：
JVM 内存分区：
```
┌─────────────────────────────────────────┐
│               堆 (Heap)                  │ ← 对象实例
│  ┌─────────────┬─────────────────────┐  │
│  │  Young Gen  │     Old Gen         │  │
│  │ (Eden+S0+S1)│                     │  │
│  └─────────────┴─────────────────────┘  │
├─────────────────────────────────────────┤
│           方法区 (Metaspace)             │ ← 类信息、常量
├─────────────────────────────────────────┤
│              栈 (Stack)                  │ ← 局部变量、方法调用
├─────────────────────────────────────────┤
│           本地方法栈                     │ ← Native 方法
├─────────────────────────────────────────┤
│           程序计数器                     │ ← 当前指令地址
└─────────────────────────────────────────┘
```

OOM 场景：
1. **堆内存溢出**：`java.lang.OutOfMemoryError: Java heap space`
   - 原因：对象太多、内存泄漏
   - 排查：dump 堆快照，分析大对象

2. **Metaspace 溢出**：`Metaspace`
   - 原因：动态生成太多类（反射、CGLIB）
   
3. **栈溢出**：`StackOverflowError`
   - 原因：递归太深

4. **直接内存溢出**：`Direct buffer memory`
   - 原因：NIO 分配过多堆外内存

**建议**：JVM 是 Java 面试必考，内存模型 + OOM 场景要熟练。

### Q9: Spring Boot 的自动装配原理是什么？

**答案**：
**核心注解**：`@SpringBootApplication` = `@SpringBootConfiguration` + `@EnableAutoConfiguration` + `@ComponentScan`

**自动装配流程**：
```
1. @EnableAutoConfiguration 触发
    ↓
2. 加载 META-INF/spring.factories（或 spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports）
    ↓
3. 获取所有 AutoConfiguration 类
    ↓
4. 根据 @Conditional 条件判断是否加载
    ↓
5. 满足条件的配置类被加载，其中的 @Bean 被注册到容器
```

**@Conditional 系列**：
- `@ConditionalOnClass`：类路径有某个类时生效
- `@ConditionalOnMissingBean`：容器中没有某个 Bean 时生效
- `@ConditionalOnProperty`：配置属性满足条件时生效

**建议**：自动装配是 Spring Boot 的核心机制，要能画出流程。

## 三、数据库（深度题）

### Q10: MySQL B+ 树索引的原理是什么？为什么用 B+ 树而不是 B 树？

**答案**：
**B+ 树特点**：
- 非叶子节点只存索引，不存数据
- 叶子节点用链表连接，支持范围查询
- 所有数据都在叶子节点

**为什么用 B+ 树**：
| 维度 | B 树 | B+ 树 |
|------|-----|-------|
| **数据位置** | 所有节点 | 只在叶子 |
| **范围查询** | 需要中序遍历 | 叶子链表顺序扫描 |
| **磁盘 IO** | 较多 | 较少（非叶子节点更小，一次 IO 能读更多索引） |

**一页能存多少索引**：
- InnoDB 页大小 16KB
- B+ 树非叶子节点只存 key + 指针（约 12 字节）
- 一页能存 16KB / 12B ≈ 1300+ 个索引
- 3 层 B+ 树能存 1300 × 1300 × 16 = 2700 万条记录

**建议**：B+ 树是 MySQL 索引的核心，要能画出结构并解释优势。

### Q11: 什么是 MVCC？解决什么问题？

**答案**：
**MVCC（Multi-Version Concurrency Control）**：多版本并发控制，用于实现读写不阻塞。

**核心机制**：
1. **隐藏列**：每行数据有 `trx_id`（事务ID）和 `roll_pointer`（回滚指针）
2. **undo log**：存储数据的历史版本
3. **Read View**：事务开始时创建，决定能看到哪些版本

**Read View 判断规则**：
```
当前事务ID: current_trx_id
Read View: [min_trx_id, max_trx_id, active_trx_ids]

如果 row.trx_id < min_trx_id → 可见（事务已提交）
如果 row.trx_id > max_trx_id → 不可见（事务在 Read View 后开始）
如果 row.trx_id in active_trx_ids → 不可见（事务未提交）
否则 → 可见
```

**解决的问题**：
- 读写不阻塞（读操作不加锁）
- 实现不同隔离级别（RC: 每次读创建新 Read View；RR: 只在事务开始时创建）

**建议**：MVCC 是 MySQL 事务的核心，要能解释 Read View 判断规则。

### Q12: MySQL 事务隔离级别有哪些？各有什么问题？

**答案**：
| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| **Read Uncommitted** | ✅ | ✅ | ✅ |
| **Read Committed** | ❌ | ✅ | ✅ |
| **Repeatable Read** | ❌ | ❌ | ✅ (InnoDB 通过 Gap Lock 解决) |
| **Serializable** | ❌ | ❌ | ❌ |

问题解释：
- **脏读**：读到其他事务未提交的数据
- **不可重复读**：同一事务内两次读取结果不同（其他事务 UPDATE）
- **幻读**：同一事务内两次查询行数不同（其他事务 INSERT）

InnoDB 默认 **Repeatable Read**，通过 **Next-Key Lock（Record Lock + Gap Lock）** 解决幻读。

**建议**：隔离级别是 MySQL 必考题，要能说清楚每个级别的问题和解决方案。

## 四、中间件（深度题）

### Q13: Redis 缓存穿透/击穿/雪崩分别是什么？怎么解决？

**答案**：
| 问题 | 场景 | 解决方案 |
|------|------|---------|
| **穿透** | 查询不存在的数据，每次都打到 DB | 布隆过滤器、缓存空值 |
| **击穿** | 热点 key 过期，大量请求打到 DB | 互斥锁、逻辑过期、热点永不过期 |
| **雪崩** | 大量 key 同时过期 | 过期时间加随机值、集群部署 |

**我在项目中的应用**：
- 客服推荐系统用了**逻辑过期**：缓存永不过期，后台异步刷新
```java
if (cache.refreshAt < now) {
    // 返回旧数据（保证可用）
    // 异步刷新
    asyncRefresh(key);
}
return cache.data;
```

**建议**：缓存三大问题是 Redis 必考，要有实际解决方案。

### Q14: Redis 分布式锁怎么实现？有什么问题？

**答案**：
**基本实现**：
```java
// 加锁
Boolean locked = redisTemplate.opsForValue()
    .setIfAbsent(lockKey, clientId, 30, TimeUnit.SECONDS);

// 解锁（Lua 脚本保证原子性）
String script = "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                "return redis.call('del', KEYS[1]) else return 0 end";
redisTemplate.execute(script, Collections.singletonList(lockKey), clientId);
```

**问题与解决**：
| 问题 | 原因 | 解决方案 |
|------|------|---------|
| **锁过期但业务未完成** | 业务执行时间超过锁的过期时间 | 看门狗自动续期（Redisson） |
| **Redis 主从切换丢锁** | 主节点挂了，锁还没同步到从节点 | RedLock（多节点加锁） |
| **非原子性解锁** | 判断和删除不是原子操作 | Lua 脚本 |

**Redisson 实现**：
```java
RLock lock = redissonClient.getLock("myLock");
lock.lock();  // 自动续期
try {
    // 业务逻辑
} finally {
    lock.unlock();
}
```

**建议**：分布式锁是高频面试题，要知道问题和解决方案。

### Q15: RabbitMQ 消息丢失和重复消费怎么解决？

**答案**：
**消息丢失的三个环节**：
1. **生产者→MQ**：Publisher Confirm
2. **MQ 存储**：持久化（durable queue + persistent message）
3. **MQ→消费者**：手动 ACK

**重复消费**：
- 原因：消费者处理完但 ACK 前宕机，MQ 重新投递
- 解决：**幂等性设计**
```java
// 方案 1：唯一 ID + Redis 去重
if (redis.setnx("msg:" + msgId, 1)) {
    // 第一次消费，处理业务
} else {
    // 重复消息，跳过
}

// 方案 2：数据库唯一约束
// INSERT INTO ... ON DUPLICATE KEY UPDATE
```

**建议**：MQ 可靠性是分布式系统的基础，要能完整回答。
