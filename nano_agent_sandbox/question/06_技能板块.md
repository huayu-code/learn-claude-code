## 一、Agent 技术栈

### Q1: 解释一下 ReAct 循环的原理？和 Chain-of-Thought 有什么区别？

**答案**：
- **Chain-of-Thought (CoT)**：LLM 一次性推理得出答案，中间步骤只是"想"不"做"
- **ReAct**：将推理（Reasoning）和行动（Acting）交替进行——先想该做什么 → 执行动作 → 观察结果 → 再想下一步。核心区别是 ReAct 可以**和外部环境交互**（调工具、执行代码），而 CoT 纯内部推理

```
CoT:  问题 → 想法1 → 想法2 → 想法3 → 答案
ReAct: 问题 → 想法1 → 执行工具 → 观察结果 → 想法2 → 执行工具 → 观察结果 → 答案
```

**建议**：ReAct 的论文是 Yao et al. 2022，面试时能提到论文名加分。

### Q2: Function Calling / Tool Calling 的原理是什么？LLM 怎么知道该调哪个工具？

**答案**：
1. 在 API 调用时，将工具的 JSON Schema（名称、描述、参数定义）作为 `tools` 参数传给 LLM
2. LLM 根据用户问题 + 工具描述，判断是否需要调用工具以及调用哪个
3. 如果需要，LLM 返回 `tool_calls`（工具名 + 参数 JSON），而非纯文本
4. 客户端执行工具 → 将结果以 `tool` 角色消息追加到上下文 → 再次调 LLM
5. LLM 看到工具结果后生成最终回复

LLM "知道该调哪个工具"的本质是**它在训练时学习了大量的工具使用模式**，加上工具描述中的自然语言描述帮助它做出选择。

**建议**：强调 Function Calling 不是魔法，是 LLM 训练 + 工具描述共同作用的结果。

### Q3: 什么是 RAG？和 Fine-tuning 相比优劣是什么？

**答案**：
**RAG（Retrieval-Augmented Generation）**：检索增强生成。将外部知识检索结果作为上下文输入给 LLM，让它基于检索内容回答。

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新 | 实时（更新检索库即可） | 需要重新训练 |
| 成本 | 低（不需要 GPU 训练） | 高（需要训练数据 + GPU） |
| 准确性 | 依赖检索质量 | 模型内化知识，更稳定 |
| 适用场景 | 知识频繁变化、领域特定问答 | 风格/格式统一、特定任务 |
| 上下文限制 | 受 token 限制 | 无（知识在模型权重中） |

**建议**：面试官常追问"什么时候用 RAG 什么时候用 Fine-tuning"——回答：知识频繁变化用 RAG，任务模式固定用 Fine-tuning，也可以两者结合。

### Q4: Prompt Engineering 有哪些常用技巧？

**答案**：
1. **Few-shot**：给 2-3 个示例，让 LLM 学习输出格式
2. **Chain-of-Thought**：提示"请一步步思考"，提升推理质量
3. **角色设定**：`You are a senior Python developer...`，约束回答风格
4. **结构化输出约束**：要求输出 JSON/XML 格式，便于程序解析
5. **负向约束**：`Do NOT include...`，避免不想要的内容
6. **分步指令**：将复杂任务拆分为明确的步骤
7. **温度控制**：`temperature=0` 提高确定性，`temperature=0.7` 增加创造性

**建议**：不需要背全，说出 4-5 个常用的即可。

### Q5: 什么是 MCP（Model Context Protocol）？你了解多少？

**答案**：
MCP 是 Anthropic 提出的一个开放协议，旨在标准化 LLM 应用与外部数据源/工具之间的通信方式。核心思想是将工具/资源的接入方式统一为标准协议，类似于 HTTP 对 Web 的标准化。

核心概念：
- **Server**：提供工具或资源的服务
- **Client**：LLM 应用或 Agent
- **Tool**：服务端暴露的可调用功能
- **Resource**：服务端暴露的数据资源

好处：
- 一次接入，多个 LLM 应用复用
- 工具开发者和 Agent 开发者解耦
- 生态标准化，类似 USB 接口

**建议**：MCP 是 2024 年的新概念，能说出来说明你关注前沿。

### Q6: 什么是 Skill？在 Agent 系统中的作用？

**答案**：
Skill 是 Agent 的"可插拔能力模块"。一个 Skill 包含：
- **知识**：该领域的专业知识和规范
- **工作流（SOP）**：标准化的操作流程
- **工具/脚本**：可执行的代码或命令

Agent 在处理任务时，根据任务类型动态加载对应的 Skill，获得该领域的专业能力。类似于人类通过"学习技能"来扩展能力边界。

例子：
- PDF 处理 Skill：知道如何读取/转换 PDF + 内置 PDF 工具
- 数据库 Skill：知道 SQL 语法规范 + 内置数据库连接工具

**建议**：结合你实习中的 Agent 系统说明 Skill 的实际应用。

### Q8: JVM 内存模型是怎样的？GC 算法了解哪些？

**答案**：
内存区域：
- **堆**：对象实例，GC 主要管理区域（新生代 Eden+S0+S1 / 老年代）
- **栈**：线程私有，局部变量表 + 操作数栈 + 动态链接
- **方法区/元空间**：类信息、常量池、静态变量
- **程序计数器**：当前线程执行的字节码行号

GC 算法：
- **标记-清除**：有碎片
- **标记-复制**：新生代使用（Eden→Survivor）
- **标记-整理**：老年代使用
- **G1**：分 Region，兼顾吞吐和延迟
- **ZGC**：超低延迟（<10ms），基于着色指针

**建议**：Java 基础面试的必考内容，每个 GC 算法能说出优劣。

### Q9: Spring Boot 的核心原理？自动装配是怎么实现的？

**答案**：
Spring Boot 自动装配原理：
1. `@SpringBootApplication` 包含 `@EnableAutoConfiguration`
2. `@EnableAutoConfiguration` 通过 `@Import(AutoConfigurationImportSelector.class)` 导入配置
3. `AutoConfigurationImportSelector` 读取 `META-INF/spring.factories`（或 Spring Boot 3 的 `AutoConfiguration.imports`）
4. 根据 `@Conditional` 系列注解（`@ConditionalOnClass`、`@ConditionalOnProperty` 等）决定哪些配置类生效
5. 生效的配置类向容器注册 Bean

核心：**约定大于配置**——通过 classpath 中的 jar 依赖自动推断需要的配置。

**建议**：Spring Boot 面试必考，自动装配流程要能画出来。

## 三、数据库

### Q13: B+ 树索引的原理？为什么 MySQL 用 B+ 树而不是 B 树？

**答案**：
B+ 树 vs B 树的区别：
- **数据存储位置**：B+ 树数据只存在叶子节点，B 树所有节点都存数据
- **叶子节点链表**：B+ 树叶子节点之间有双向链表，支持范围查询
- **扇出更高**：B+ 树非叶子节点不存数据，同一页可以放更多 key，树更矮，IO 更少

MySQL 选 B+ 树的原因：
1. **范围查询友好**：`WHERE age > 18` 只需找到起点然后沿链表扫描
2. **IO 效率高**：树更矮意味着更少的磁盘 IO（每层 = 一次磁盘读取）
3. **全表扫描效率**：直接遍历叶子节点链表，不需要中序遍历整棵树

**建议**：B+ 树是数据库面试第一题，必须能画出来并解释范围查询的过程。

### Q14: MySQL 的事务隔离级别有哪些？MVCC 怎么实现的？

**答案**：
四种隔离级别：
1. **Read Uncommitted**：脏读 ✓ 不可重复读 ✓ 幻读 ✓
2. **Read Committed**：脏读 ✗ 不可重复读 ✓ 幻读 ✓（每次读创建新 ReadView）
3. **Repeatable Read**：脏读 ✗ 不可重复读 ✗ 幻读 ✗（InnoDB 默认，一个事务一个 ReadView）
4. **Serializable**：全部 ✗（串行执行）

MVCC 实现：
- 每行有隐藏字段：`DB_TRX_ID`（最后修改的事务 ID）、`DB_ROLL_PTR`（回滚指针，指向 undo log）
- **ReadView**：记录当前活跃的事务 ID 列表，用于判断某个版本对当前事务是否可见
- 读取时沿 undo log 链找到第一个对当前 ReadView 可见的版本

**建议**：MVCC 是 MySQL 面试的深水区，能画出 undo log 链 + ReadView 判断逻辑就过关了。

### Q15: SQL 调优你一般怎么做？

**答案**：
1. **EXPLAIN 分析**：看 type（ALL→index→range→ref→const，越右越好）、rows、Extra
2. **索引优化**：
   - 覆盖索引：避免回表
   - 联合索引：最左前缀匹配
   - 避免索引失效：不在索引列上做函数/计算、不用 `!=`/`NOT IN`
3. **查询优化**：
   - 避免 `SELECT *`
   - 大分页用 `WHERE id > last_id LIMIT n`（游标分页）
   - 子查询改 JOIN
4. **慢查询日志**：开启 `slow_query_log`，找出耗时 SQL

**建议**：SQL 调优面试不会很深，能说出 EXPLAIN + 索引优化 + 分页优化即可。

## 四、中间件

### Q16: Redis 缓存穿透/击穿/雪崩分别是什么？怎么解决？

**答案**：
| 问题 | 描述 | 解决方案 |
|------|------|---------|
| **穿透** | 查询一定不存在的数据（如 id=-1），请求直达数据库 | 布隆过滤器 / 缓存空值 |
| **击穿** | 热点 key 过期瞬间，大量并发请求打到数据库 | 互斥锁 / 逻辑过期（不设TTL） |
| **雪崩** | 大量 key 同时过期，数据库被压垮 | 过期时间加随机值 / 多级缓存 |

你在简历中实际用到了：
- **推荐系统**用了"逻辑过期"方案防击穿
- **客服系统**的 RabbitMQ 异步更新本质是"写后更新缓存"防不一致

**建议**：三者的区别和解决方案是 Redis 面试必考题，能结合项目说明更好。

### Q17: Redis 分布式锁的原理？Redisson 的 WatchDog 机制？

**答案**：
基本原理：`SET key value NX PX timeout`
- NX：不存在才设置（保证互斥）
- PX：设过期时间（防死锁）

Redisson WatchDog：
- 问题：如果业务执行时间 > 锁的过期时间，锁会提前释放，其他线程拿到锁，出现并发问题
- 解决：WatchDog 后台线程每隔 `lockWatchdogTimeout/3`（默认 10s）自动续期
- 续期条件：持锁线程仍然存活 && 未主动释放
- 线程结束/异常：WatchDog 停止续期 → 锁自动过期释放

**建议**：WatchDog 的续期逻辑是 Redisson 面试的核心考点。

---

### Q18: RabbitMQ 如何保证消息不丢失？

**答案**：
三段保障（和客服系统项目中的回答一致）：
1. **生产端**：Publisher Confirm（Broker 收到消息后回调确认）
2. **Broker 端**：消息持久化（durable queue + persistent message）
3. **消费端**：手动 ACK（处理成功后才确认，失败则 reject + 重入队列或进死信）

**建议**：这是 RabbitMQ 面试的标准三段式回答，所有 MQ 面试都适用。

## 五、工具

### Q19: Git 的常用操作？rebase 和 merge 的区别？

**答案**：
- **merge**：创建一个新的合并提交，保留两个分支的完整历史。结果是菱形/钻石形状的提交图
- **rebase**：将当前分支的提交"搬到"目标分支的末尾，形成线性历史。更干净但改写了提交历史

使用场景：
- 个人开发分支 → `rebase`（保持历史整洁）
- 团队协作/公共分支 → `merge`（不改写历史，安全）

**建议**：Git 面试不会太深，merge vs rebase 是最常问的。

### Q20: Docker 的核心原理？容器和虚拟机的区别？

**答案**：
Docker 核心技术：
- **Namespace**：进程/网络/文件系统等隔离（PID/NET/MNT/UTS/IPC/USER）
- **Cgroup**：资源限制（CPU/内存/IO）
- **Union FS**：分层镜像（overlay2）

容器 vs 虚拟机：
| 维度 | 容器 | 虚拟机 |
|------|------|--------|
| 隔离层 | 共享主机内核 | 独立内核（Hypervisor） |
| 启动速度 | 秒级 | 分钟级 |
| 资源开销 | 低（MB 级） | 高（GB 级） |
| 安全性 | 较低（共享内核） | 高（完全隔离） |

**建议**：Docker 面试常和你的沙箱项目关联——"为什么沙箱不用 Docker"可以接着聊。

### Q21: OpenTelemetry 是什么？和 Prometheus + Grafana 的关系？

**答案**：
- **OpenTelemetry**：统一的可观测性标准（CNCF 项目），定义了 Traces/Metrics/Logs 的采集、传输、导出规范。它是"采集层"的标准
- **Prometheus**：时序数据库，存储和查询 Metrics
- **Grafana**：可视化面板，展示 Prometheus 等数据源的图表
- **关系**：OpenTelemetry 采集数据 → 导出到 Prometheus（Metrics）/ Jaeger（Traces） → Grafana 展示

**建议**：可观测性是云原生面试的常见考点，能说清楚三者的分工即可。

### Q22: Langfuse 和 OpenTelemetry 的区别？为什么 LLM 应用需要专门的可观测性工具？

**答案**：
OpenTelemetry 是通用的分布式追踪，关注的是**系统层面**（RPC 耗时、错误率、调用链路）。但 LLM 应用有特殊的观测需求：
- **Token 消耗追踪**：每次调用消耗多少 token、成本是多少
- **Prompt 版本管理**：哪个 Prompt 版本效果更好
- **模型输出质量**：回答准确率、Tool Calling 成功率
- **Agent 轨迹**：多轮工具调用的完整决策路径

Langfuse 专门解决这些 LLM 特有的观测需求，和 OpenTelemetry 互补而非替代。

**建议**：这个问题直接关联你的实习经历，说清楚"为什么两个都要"。
