# 实习经历 — AI PPT 智能生成系统 面试题

---

## 一、整体架构理解（了解性）

### Q1: AI PPT 生成系统的整体架构是什么？用户从输入主题到拿到 PPT 的完整流程？

**答案**：
```
用户输入主题（如"Q3 营销策划方案"）
  → LLM 解析意图 + 搜索相关素材
  → LLM 生成大纲（10 页标题 + 每页要点）
  → 模板语义检索，匹配最佳模板
  → 逐页生成 JSX 内容
  → 逐页并行渲染（JSX 校验 → 图片处理 → 缓存）
  → 批量合成最终 PPT 文件
  → 通过 Kaiwu 沙箱交付给用户
```

核心架构是 **LLM Agent + Tool Calling**：LLM 作为"大脑"自主决策调用哪些工具、按什么顺序调用，10+ 个工具作为"手脚"执行具体操作。

**建议**：画图说明最直观。重点是让面试官理解"LLM 不是一次性生成 PPT，而是通过多轮工具调用逐步构建"。

---

### Q2: 为什么用 LLM Agent + Tool Calling 而不是传统的固定 Pipeline？

**答案**：
- **固定 Pipeline**：步骤写死（搜索→大纲→逐页→合成），无法处理异常（如某一页图片搜不到、某一页 JSX 校验失败）
- **Agent 架构**：LLM 根据每一步的执行结果**自主决定下一步**，遇到图片搜不到可以换关键词重搜，JSX 校验失败可以修改后重试
- 举例：用户说"帮我做一个关于 AI 的 PPT，第 3 页要用我上传的图片"——Agent 能理解这个复合需求，先处理其他页面，再单独处理第 3 页的用户图片

**建议**：用一个具体的异常处理场景来说明 Agent 架构的优势，比干巴巴说"灵活性"更有说服力。

---

### Q3: 10+ 个工具具体有哪些？

**答案**：
```
主题搜索工具     — 联网搜索主题相关素材/数据
大纲生成工具     — 生成 PPT 大纲（页数 + 每页标题 + 要点）
大纲设定工具     — 确认/修改大纲
JSX 写入工具     — 为每一页生成 JSX 代码
JSX 校验工具     — 远程校验 JSX 语法和渲染可行性
图片搜索工具     — 根据关键词搜索图片（每次搜 10 张）
图片选择工具     — 从 10 张中智能选择最匹配的
图片生成工具     — 调用混元生图，生成自定义图片
图片筛选工具     — 按形状/分辨率筛选（横版/竖版/方形）
页面缓存工具     — Redis HSET 缓存每页渲染状态
批量合成工具     — 将所有页面合成为最终 PPT 文件
模板检索工具     — 向量化语义搜索匹配模板
```

**建议**：不需要背全部名字，但至少能说出 8-9 个核心工具，体现你对整个生成链路的掌握。

## 二、逐页并行渲染管线（核心亮点 — 深度题）

### Q4: "逐页并行渲染"是怎么实现的？10 页是同时渲染还是有依赖关系？

**答案**：
- 大纲确定后，10 页的内容生成是由 LLM **逐页生成**的（因为后页可能参考前页内容）
- 但每一页的**渲染流程**（JSX 校验 → 图片处理 → 缓存）是独立的，可以并行
- 实现方式：LLM 生成一页 JSX 后立即触发该页的渲染 goroutine，不等渲染完成就继续生成下一页
- 最终所有页面渲染完成后，触发批量合成

这是一个 **"流水线并行"** 模式：LLM 生成和页面渲染两个阶段重叠执行。

**建议**：面试官关心的是"你对并行的理解不只是多个 goroutine 跑同一个函数，而是不同阶段的 pipeline 并行"。

---

### Q5: 图片处理流程中"搜图 10 张智能选图"是怎么做的？选图的标准是什么？

**答案**：
1. 根据每页内容提取关键词，调图片搜索 API 获取 10 张候选图片
2. **智能选图**：将 10 张图片的描述/标签 + 当前页面内容一起发给 LLM，让 LLM 选择最匹配的 1-2 张
3. **形状筛选**：根据 PPT 模板的图片占位符形状（横版 16:9 / 竖版 9:16 / 方形 1:1），筛选分辨率匹配的图片
4. **分辨率筛选**：过滤掉低于最小分辨率要求的图片，避免拉伸模糊

如果搜图没有合适的结果，降级为调用混元生图工具，根据页面内容描述生成一张图片。

**建议**：这个流程体现了"多路并行 + 降级兜底"的工程思维。

---

### Q6: Redis HSET 缓存页面状态是怎么设计的？

**答案**：
```
Key:   ppt:{task_id}
Field: page_1, page_2, ... page_10
Value: {status: "rendering/completed/failed", jsx: "...", images: [...], rendered_at: "..."}
```

用 HSET 而非多个 String key 的原因：
- **原子性**：一个 HSET 对应一个 PPT 任务，`HGETALL` 一次获取所有页面状态
- **内存效率**：Redis 对小 HSET 有 ziplist 优化
- **过期管理**：整个 HSET 设统一 TTL（如 1 小时），不需要管理 10 个独立 key 的过期

**建议**：Redis 数据结构选型是面试高频题，说清楚"为什么用 HSET 而不是 String"。

---

### Q7: 单次生成 10 页 PPT 耗时 < 2 分钟是怎么做到的？耗时瓶颈在哪？

**答案**：
耗时拆解：
- LLM 生成大纲：~10s
- 模板检索：~2s
- LLM 逐页生成 JSX：~30-40s（10 页，流水线并行）
- 图片处理（并行）：~20-30s（搜图 + 选图 + 下载，这是最大瓶颈）
- JSX 校验（并行）：~5-10s
- 批量合成：~5s
- 总计：~70-90s < 2 分钟

**瓶颈**在图片处理，因为涉及外部 API 调用（图片搜索、图片下载）。优化措施：
- 10 张候选图片的元信息获取并行
- 图片下载和 JSX 校验并行（不互相依赖）
- 如果图片搜索超时，直接用混元生图兜底

**建议**：面试官最爱问"耗时瓶颈在哪"，要能做出这样的拆解分析。

## 三、模板向量化检索（深度题）

### Q8: 模板语义检索系统是怎么构建的？

**答案**：
1. **离线阶段**：
   - 收集所有 PPT 模板的元信息（风格、行业、颜色主题、适用场景等）
   - 将元信息文本通过混元 Embedding 模型转为 1024 维向量
   - 向量存入 Elasticsearch 的 dense_vector 字段

2. **在线阶段**：
   - 用户输入主题 + LLM 生成的大纲关键词，拼接为查询文本
   - 通过混元 Embedding 转为查询向量
   - Elasticsearch 做 cosine similarity 搜索，返回 Top-K 匹配模板
   - LLM 从 Top-K 中做最终选择

**建议**：这是标准的向量检索架构（Embedding + Vector DB + ANN Search），面试官可能追问 Embedding 模型细节。

---

### Q9: 为什么用 Elasticsearch 做向量搜索而不是专业的向量数据库（如 Milvus、Pinecone）？

**答案**：
- **已有基础设施**：团队已有 ES 集群，不需要额外运维新组件
- **混合搜索**：ES 可以同时做关键词搜索（BM25）+ 向量搜索（dense_vector），两路结果融合效果更好
- **数据规模**：模板数量在万级别，ES 的向量搜索完全够用（百万级以上才需要专业向量数据库）
- **权衡**：ES 的 ANN 搜索精度和性能不如 Milvus，但在模板检索场景下差距可忽略

**建议**：技术选型一定要说清楚"为什么在当前场景下够用"，而非"哪个更好"。

---

### Q10: 模板匹配准确率提升 40% 是怎么测的？

**答案**：
- **对照组**：关键词搜索（用户输入主题 → ES 的 BM25 全文搜索 → 按相关性排序）
- **实验组**：向量语义搜索（主题 → Embedding → cosine similarity）
- **评测方式**：人工标注 200 个测试用例（主题 → 最佳模板），分别用两种方案检索，计算 Top-3 命中率
- 关键词搜索：Top-3 命中率 ~55%
- 语义搜索：Top-3 命中率 ~77%
- 提升约 (77-55)/55 ≈ 40%

**建议**：如果没有做过正式评测，可以说"通过内部测试用例评估"，但要能说清楚评测方法。

## 四、多模型支持（了解性）

### Q11: 支持 5 种模型（GPT/Claude/Gemini/DeepSeek/混元），为什么需要按模型切换 Prompt？

**答案**：
不同模型对 Prompt 的响应差异很大：
- **GPT-4**：对结构化指令（JSON schema 约束）遵循度高，Prompt 可以更精简
- **Claude**：对 XML 标签格式响应好，用 `<page>` `<content>` 等 XML 包裹效果更佳
- **混元**：中文理解更好，但对复杂 JSON 输出的稳定性不如 GPT，需要更多 few-shot 示例
- **DeepSeek**：性价比高但指令遵循度稍弱，需要更明确的约束

统一 Prompt 会导致某些模型表现明显下降，按模型维护 Prompt 模板是 LLM 应用的常见实践。

**建议**：这个问题体现你对不同 LLM 特性的实战理解，比只说"调过 API"更有深度。

---

### Q12: Kaiwu 沙箱是什么？为什么 PPT 生成需要沙箱？

**答案**：
Kaiwu 沙箱是 AI PPT 项目使用的执行环境（和我第一个项目的 AGS 沙箱类似但是不同的平台）。PPT 生成需要沙箱的原因：
- **JSX 渲染**：JSX 代码需要在 Node.js 环境中渲染为图片/组件，不能在主服务进程中执行（安全 + 隔离）
- **产物生成**：最终 PPT 文件的组装需要用到 pptx 库等第三方依赖，沙箱中预装了这些环境
- **文件交付**：生成的 PPT 文件存在沙箱文件系统中，通过 URL 交付给用户下载

**建议**：说清楚"为什么不在主服务中直接做"——安全隔离 + 依赖隔离。

## 五、设计决策 & 追问（深度题）

### Q13: 如果某一页的渲染失败了（JSX 校验不通过），整个 PPT 怎么处理？

**答案**：
不会废弃整个 PPT，而是做局部重试和降级：
1. **自动重试**：将校验错误信息反馈给 LLM，让 LLM 修改 JSX 后重新校验（最多重试 2-3 次）
2. **降级模板**：如果重试仍失败，使用该页的简化模板（纯文本 + 简单布局），保证内容不丢失
3. **标记异常**：在最终结果中标记该页为"自动降级"，通知用户可手动调整

这样 10 页中即使 1-2 页出问题，其他 8-9 页正常的 PPT 仍可交付。

**建议**：这体现了"部分失败不影响整体"的容错设计，是生产系统的核心要求。

---

### Q14: JSX→PPT 和 Markdown→PPT 双链路分别适用什么场景？

**答案**：
- **JSX→PPT**：适合对视觉效果要求高的场景——支持自定义布局、图片定位、样式控制，但 LLM 生成 JSX 的难度更高、出错率也更高
- **Markdown→PPT**：适合对内容要求高但视觉简洁的场景——Markdown 结构简单、LLM 生成稳定性好，但布局灵活性有限
- 实际使用中默认走 JSX 链路（效果更好），当 JSX 链路连续失败时降级为 Markdown 链路

**建议**：双链路的设计本质是"效果 vs 稳定性"的权衡，高质量优先但有降级保底。

---

### Q15: 如果让你优化这个系统，你会从哪些方面入手？

**答案**：
1. **生成速度**：引入流式生成——一边生成一边展示已完成的页面，不用等全部完成（提升用户感知速度）
2. **图片质量**：建立图片质量评分模型，自动过滤低质量图片，减少人工选图的需要
3. **用户反馈闭环**：收集用户对每页的修改记录，作为 fine-tuning 数据优化 LLM 的 PPT 生成能力
4. **成本优化**：对"简单页面"（如目录页、致谢页）用轻量模型生成，只对"核心页面"用 GPT-4 级模型

**建议**：改进方向要结合具体的痛点，不要说"加缓存""加集群"这种通用回答。
