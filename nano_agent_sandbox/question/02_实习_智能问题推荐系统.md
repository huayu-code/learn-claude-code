# 实习经历 — 智能问题推荐系统 面试题

---

## 一、需求理解（了解性）

### Q1: 这个推荐系统解决什么业务问题？用户场景是什么？

**答案**：
用户打开一篇文档并唤起 AI 助手时，面对空白对话框不知道"能问什么/怎么用"。这是典型的**冷启动问题**——用户不了解 AI 的能力边界，导致使用率低。推荐系统通过分析用户最近的文档内容，主动推荐 3 条个性化问题，引导用户开始对话，降低使用门槛。

**建议**：先讲业务痛点，再讲方案，面试官更容易理解。

---

### Q2: 为什么选择基于用户文档来生成推荐，而不是基于热门问题或用户历史行为？

**答案**：
- **热门问题**：对所有用户一样，缺乏个性化，用户觉得"和我无关"
- **用户历史行为**：新用户没有历史行为，仍然有冷启动问题
- **基于文档**：文档是用户当前正在处理的内容，推荐与文档相关的问题**上下文相关性最强**，用户感知是"AI 懂我在做什么"
- 实际效果：基于文档的推荐点击率比固定推荐提升约 **30%**

**建议**：如果面试官追问"有没有做过 A/B 测试"，要如实回答。30% 的数据如果是 A/B 实验结果，含金量很高；如果是估算，要说明估算方式。

---

## 二、技术实现（深度题）

### Q3: "自动获取用户最近 20 篇文档"——为什么是 20 篇？支持 6 种品类是哪 6 种？

**答案**：
- **20 篇**：是在"信息量"和"处理耗时"之间的权衡。太少（如 5 篇）可能无法覆盖用户近期的工作主题；太多（如 50 篇）会增加摘要提取的延迟和 LLM token 消耗。20 篇平均覆盖用户近 1-2 周的活跃文档
- **6 种品类**：文档（Doc）、表格（Sheet）、演示文稿（Slides）、思维导图（MindMap）、流程图（Flowchart）、收藏（Collection）等（根据实际情况调整）

**建议**：数字的选择一定要能解释 trade-off，不能只说"拍脑袋定的"。

---

### Q4: "通过 trpc.GoAndWait 并发提取正文摘要"——这个并发怎么实现的？为什么用 GoAndWait？

**答案**：
- `trpc.GoAndWait` 是 tRPC-Go 框架提供的并发原语，类似 `sync.WaitGroup` + `errgroup.Group`，可以并发执行多个 RPC 调用并等待全部完成
- 每篇文档的摘要提取是一次独立的 RPC 调用（获取文档内容 + 截取前 200 字），20 篇文档串行需要 ~2-4s，并发后 ~200-500ms（取决于最慢的那个）
- 同时并发获取每篇文档的权限信息（编辑/只读），与摘要提取并行

**建议**：面试官可能追问"如果某一篇文档的 RPC 超时怎么办"——回答：单个失败不影响整体，跳过该文档，最终可能只用 18-19 篇。

---

### Q5: 前 200 字作为摘要够吗？为什么不用全文？

**答案**：
- **token 成本**：20 篇文档 × 全文 = 可能几十万 token，一次推荐的成本太高
- **信息密度**：文档的前 200 字通常包含标题、摘要、引言，信息密度最高
- **LLM 上下文限制**：20 篇 × 200 字 = 4000 字，加上 Prompt 模板和输出空间，总共控制在 8K token 以内
- **权衡**：确实可能丢失文档后半部分的关键信息，但在推荐场景下前 200 字足够确定文档主题

**建议**：如果面试官说"200 字太少"，可以反驳"这是推荐场景不是精读场景，我们要的是主题识别而非深度理解"。

---

### Q6: 权限感知策略具体怎么实现的？"只读推荐分析类、可编辑推荐创作类"这个分类逻辑在哪做的？

**答案**：
权限信息在并发阶段一起获取，然后在构造 Prompt 时将文档按权限分类：
```
你是一个智能推荐助手。以下是用户近期的文档：

【可编辑文档】（优先推荐创作/编辑/优化类问题）：
1. 《Q3 营销策划案》- 摘要：...
2. 《产品需求文档》- 摘要：...

【只读文档】（仅推荐分析/总结/解读类问题）：
1. 《竞品分析报告》- 摘要：...
2. 《行业白皮书》- 摘要：...

请生成 15 条推荐问题...
```

分类逻辑在 **Prompt 层面**通过自然语言指导 LLM 实现，而非硬编码规则。这样更灵活，且可以通过 Langfuse 动态调整 Prompt 来优化策略。

**建议**：这个设计体现了"把分类逻辑委托给 LLM"的思想，是 AI 应用的常见模式，面试官会认可。

---

### Q7: "一次生成 15 条，每次随机抽 3 条返回"——为什么不直接生成 3 条？

**答案**：
- **多样性**：直接生成 3 条的话，用户每次看到的是同样的推荐，刷新也不会变，体验差
- **成本效率**：生成 15 条和生成 3 条的 LLM 调用成本差异不大（同一次调用），但 15 条可以支撑 5 次随机展示
- **缓存友好**：生成一次缓存 15 条，多次随机抽取，避免频繁调用 LLM
- **A/B 空间**：15 条中可以按类型标签（分析/创作/格式化等）做加权随机，为后续精细化推荐留余地

**建议**：这个设计体现了"一次计算多次使用"的工程思维，和实际生产中"离线计算 + 在线随机"的推荐架构一致。

## 三、缓存架构（深度题）

### Q8: "Redis 永久缓存 + 按小时异步刷新"是怎么实现的？为什么不设 TTL？

**答案**：
- **永久缓存（不设 TTL）**：保证任何时刻用户访问都有推荐可返回，不会因缓存过期而出现空推荐
- **按小时异步刷新**：每个缓存 key 额外存一个 `refresh_at` 时间戳，请求到来时检查：
  - 未过期：直接返回缓存（RT < 50ms）
  - 已过期：返回旧缓存（保证用户体验），同时 `trpc.Go` 异步触发刷新（不阻塞当前请求）
  - 异步刷新完成后覆盖缓存

这就是经典的 **"缓存永不过期 + 异步更新"** 模式，也叫"逻辑过期"策略。

**建议**：面试官很可能追问"这不就是缓存击穿的解决方案吗"——是的，逻辑过期是防止缓存击穿的经典方案之一。

---

### Q9: 异步刷新失败怎么办？会不会一直返回过时的推荐？

**答案**：
- 异步刷新失败会记录日志和错误指标，**不影响当前返回**（用户看到的是旧推荐，比看到空白好）
- 设计了重试机制：下一次用户访问时如果缓存仍然过期，会再次触发异步刷新
- 设置了**最大容忍过期时间**（如 24 小时），超过这个时间的缓存认为太旧，改为同步刷新（牺牲延迟保证新鲜度）
- 监控告警：如果某用户的缓存连续刷新失败超过阈值，触发告警

**建议**：这体现了"可用性优先"的设计思想——宁可返回旧数据，不可返回空数据。

---

### Q10: 缓存的 key 怎么设计的？

**答案**：
```
recommend:questions:{user_id}
```
以用户维度缓存，因为推荐是基于"用户最近的文档"生成的，同一用户在不同文档下看到的推荐可能一样（都是基于最近 20 篇文档）。

如果要做更精细的"每篇文档不同推荐"，key 可以改为 `recommend:questions:{user_id}:{doc_id}`，但会导致缓存数量膨胀。当前方案在效果和成本之间取平衡。

**建议**：key 设计看似简单但能体现你对业务的理解。面试官可能追问"换了文档推荐还一样吗"，要能解释清楚。

---

### Q11: 接口 RT < 50ms 是怎么做到的？

**答案**：
- 缓存命中时的链路：请求 → 读 Redis → 随机抽 3 条 → 返回
- Redis 读操作本身 ~1-5ms（内网）
- 随机抽取 + JSON 序列化 ~1-2ms
- 网络传输 ~10-30ms
- 总计 < 50ms

关键点：**没有任何同步的 LLM 调用在请求链路中**，所有重计算都在异步刷新时完成。

**建议**：这体现了"把重计算放到离线/异步，在线只做轻量查询"的架构思想。

## 四、Langfuse Prompt 管理（了解性）

### Q12: Langfuse 动态 Prompt 管理是怎么实现的？

**答案**：
- Langfuse 提供 Prompt Management 功能，可以在 Web UI 上编辑 Prompt 模板，通过 API 拉取
- 测试环境：代码中通过 Langfuse SDK 实时拉取最新 Prompt 版本，方便快速迭代调试
- 生产环境：锁定 Prompt 版本号，避免误修改影响线上
- 好处：Prompt 迭代不需要改代码、不需要发版、不需要走 CI/CD，产品经理/运营也能参与 Prompt 调优

**建议**：这是 LLM 应用的最佳实践，体现你对"Prompt 即配置"理念的理解。

---

### Q13: "Prompt 迭代无需发版"在实际工作中带来了什么价值？

**答案**：
推荐系统上线后，产品经理对推荐问题的质量有多次调整需求（比如"多推荐一些格式化类问题"、"减少重复感"）。如果每次都改代码发版，走一次发布流程需要 1-2 天。通过 Langfuse 动态 Prompt，产品经理在测试环境修改 Prompt → 验证效果 → 同步到生产，**全程 10 分钟**，迭代效率提升数十倍。

**建议**：用具体的业务场景来说明技术价值，比纯技术解释更有说服力。

## 五、设计权衡 & 追问（深度题）

### Q14: 推荐的点击率提升 30% 是怎么测的？

**答案**：
- 对照组：固定推荐（所有用户看到相同的 3 条通用问题，如"帮我总结"、"帮我翻译"等）
- 实验组：基于文档的个性化推荐
- 指标：推荐问题的点击率（点击数 / 展示数）
- 周期：上线后观察 1-2 周的数据对比

（如果没有做过严格 A/B 测试，可以说：）
30% 是基于上线前后的点击率对比估算，不是严格的 A/B 实验。严格来说还需要控制其他变量（如用户群体差异、时间段差异等）。

**建议**：数据真实性是面试官最关心的，不要虚构。如果是估算，诚实说明估算方式。

---

### Q15: 如果用户没有最近的文档（新用户）怎么办？

**答案**：
降级策略：
1. 先查用户最近 20 篇文档
2. 如果文档数 < 5 篇（信息不足以生成个性化推荐），降级为**品类推荐**——根据用户所在的产品品类推荐通用问题
3. 如果连品类信息都没有（纯新用户），降级为**热门推荐**——展示全平台点击率最高的 3 条通用问题

三级降级保证任何用户都有推荐可看。

**建议**：这个问题很常见，准备好降级方案说明你考虑了边界情况。

---

### Q16: 这个系统的瓶颈在哪？如果用户量增长 10 倍怎么办？

**答案**：
- **瓶颈 1：LLM 调用**——生成 15 条推荐需要 ~3-5s，但因为是异步刷新，不在请求链路中，对用户无感
- **瓶颈 2：Redis 容量**——每个用户约 2-3KB 缓存（15 条推荐 JSON），100 万用户 = ~3GB，Redis 可承受
- **10 倍增长应对**：
  - Redis 可以加从节点分担读压力
  - LLM 异步刷新可以加队列控速，避免突发大量并发调用
  - 进一步延长刷新间隔（如 2 小时），降低 LLM 调用频次

**建议**：面试官考的是你的系统设计思维——能不能做容量预估、能不能找到瓶颈、有没有扩展方案。
