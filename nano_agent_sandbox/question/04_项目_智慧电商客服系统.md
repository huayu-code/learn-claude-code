# 项目经历 — 智慧电商客服系统 面试题

---

## 一、整体架构理解（了解性）

### Q1: 简单介绍一下这个客服系统的架构和你做了什么？

**答案**：
这是一个基于微服务架构的智能客服系统，核心能力是让 AI 自动回答用户的客服咨询，并能通过 Tool Calling 执行实际操作（查订单、查物流等）。我主要负责三块：
1. **RAG 问答系统**：ES 向量库 + 大模型，回答商品/政策类问题
2. **Tool Calling 集成**：让 AI 能调用内部接口执行操作
3. **性能优化**：已读状态的 Redis Bitmap 存储 + 异步批量同步

**建议**：先给全景再说细节。面试官最关心的通常是 RAG 和 Tool Calling 部分，性能优化是加分项。

---

## 二、RAG 系统（深度题）

### Q2: RAG 系统的完整链路是什么？从用户提问到回答的全流程？

**答案**：
```
用户提问（如"这个商品能退货吗"）
  → 文本向量化（Spring AI 内置 Embedding）
  → ES 向量搜索（cosine similarity，Top-K 召回）
  → 拼接上下文（检索到的文档片段 + 用户问题）
  → 发送给大模型生成回答
  → 返回用户
```

**建议**：这是标准的 RAG 流程，面试官会追问每个环节的细节。

---

### Q3: ES 向量库里存的是什么数据？怎么做的数据准备？

**答案**：
存储的数据包括：
- **商品信息**：商品描述、规格参数、常见问题
- **退换货政策**：各品类的退换货规则、时间限制、条件
- **活动规则**：促销活动的参与条件、优惠计算方式

数据准备流程：
1. 从 MySQL 导出原始文档
2. 按段落切分（chunk），每个 chunk 300-500 字
3. 通过 Embedding 模型转为向量
4. 写入 ES 的 dense_vector 字段，同时保留原文用于上下文拼接

**建议**：面试官大概率追问"chunk 策略怎么定的"、"chunk 太大/太小有什么问题"。

---

### Q4: Chunk 大小怎么确定的？300-500 字有什么依据？

**答案**：
- **太小**（如 100 字）：语义不完整，检索到的片段可能缺少关键上下文，LLM 无法给出准确回答
- **太大**（如 2000 字）：向量表征被稀释（一个向量要代表太多内容），检索精度下降；且占用更多 token
- **300-500 字**：经验值，覆盖一个完整的问答对或一条完整的政策规则。实际做了对比测试，300-500 字在检索准确率和回答质量上最优

**建议**：最好能说出"做了对比测试"，即使是简单的人工评估也比"拍脑袋"好。

---

### Q5: RAG 系统有没有遇到过"检索到了但回答不对"的情况？怎么处理？

**答案**：
有，主要两类问题：
1. **检索到不相关的内容**（召回噪声）：
   - 优化：调整相似度阈值，低于阈值的结果过滤掉
   - 优化：在检索结果中加入 BM25 关键词匹配做 rerank
2. **检索到了但 LLM 没用好**（答非所问）：
   - 优化：改进 Prompt，明确要求"仅根据以下参考资料回答，如果资料中没有相关信息则说明不确定"
   - 优化：对 LLM 回答做后处理校验（关键实体是否在检索文档中出现）

**建议**：这是 RAG 的经典问题，面试官一定会问。准备好真实案例最好。

---

### Q6: ES 中同时存了向量和原始文本，搜索时是纯向量搜索还是混合搜索？

**答案**：
最初是纯向量搜索，后来改为**混合搜索**（向量 + BM25 关键词）效果更好：
- 向量搜索擅长语义相似（"退货"能匹配"退换货政策"）
- 关键词搜索擅长精确匹配（商品编号、专有名词）
- 两路结果通过 RRF（Reciprocal Rank Fusion）融合排序

例如用户问"订单号 20240101001 的物流在哪"——纯向量搜索可能匹配到通用物流政策，关键词搜索能精确匹配订单号相关文档。

**建议**：如果没做混合搜索，如实说"当前是纯向量搜索"，然后说"如果改进会加入混合搜索"。

## 三、Tool Calling（深度题）

### Q7: Tool Calling 是怎么实现的？Spring AI 提供了哪些支持？

**答案**：
Spring AI 提供了 Function Calling 的标准化接口：
1. 定义工具函数（如 `queryOrder(orderId)`, `trackLogistics(orderId)`），用 `@Description` 注解描述功能
2. 注册到 Spring AI 的 ChatClient 中
3. 用户提问时，LLM 判断是否需要调用工具：
   - 需要：返回 tool_call 指令 → Spring AI 自动路由到对应函数 → 执行 → 结果回传 LLM → 生成最终回答
   - 不需要：直接用 RAG 检索结果回答

**建议**：Spring AI 的 Function Calling 封装了 OpenAI 的 tool_use 协议，面试官可能追问底层原理。

---

### Q8: "满足 95% 的客服查询需求自动回应"——这个数据怎么来的？剩下 5% 怎么处理？

**答案**：
- **95%**：统计上线后 2 周内所有客服咨询，按类型分类：
  - 商品咨询（~40%）→ RAG 覆盖
  - 订单/物流查询（~30%）→ Tool Calling 覆盖
  - 退换货咨询（~20%）→ RAG 覆盖
  - 复杂投诉/特殊情况（~5%）→ 需要人工客服
  - 其他（~5%）→ 需要人工客服
- **剩下 5%**：AI 检测到无法处理时（如用户情绪激动、连续追问、涉及赔偿等），自动转接人工客服，并将对话历史一起传递

**建议**：数据要能自圆其说。如果是估算，说明估算逻辑。

---

### Q9: 如果 Tool Calling 调用内部接口失败怎么办（如订单服务超时）？

**答案**：
1. **超时控制**：每个工具调用设 3-5s 超时，避免长时间阻塞对话
2. **错误处理**：将错误信息作为工具结果返回给 LLM，LLM 会自动生成友好的错误提示（如"抱歉，订单系统暂时繁忙，请稍后再试"）
3. **降级回答**：如果是查询类请求，降级为引导用户到自助页面查看
4. **重试**：对于幂等的查询接口，自动重试 1 次

**建议**：这体现了"AI 系统的容错不只是技术层面，还要考虑用户体验层面"。

## 四、Redis Bitmap 已读状态（深度题）

### Q10: 为什么用 Redis Bitmap 存储已读状态？相比其他方案节约了多少？

**答案**：
已读状态本质是"用户 X 是否已读消息 Y"——一个二值状态（已读/未读）。

方案对比：
| 方案 | 存储 1 万用户 × 1 万消息 |
|------|--------------------------|
| MySQL 一行一记录 | 1 亿行，约 ~10GB |
| Redis Hash | 1 亿个 field，约 ~5GB |
| Redis Bitmap | 1 万个 key × 1.25KB = ~12.5MB |

Bitmap 将每条消息映射为一个 bit 位，1 个用户的 1 万条消息只需 1.25KB（10000/8 字节），空间节约 **98%+**。

**建议**：面试官必追问"Bitmap 的局限性"——消息 ID 必须是连续整数（或可映射为整数偏移量），如果消息 ID 是 UUID 则不适用。

---

### Q11: Bitmap 的 key 是怎么设计的？如果消息量很大怎么办？

**答案**：
```
Key:   read_status:{user_id}:{date}   （按天分片）
Offset: message_id 的偏移量
```

按天分片的原因：
- 单个 Bitmap 太大会影响操作性能（GETBIT/SETBIT 虽然是 O(1)，但大 key 的网络传输和内存分配有开销）
- 按天分片后，过期的天可以直接删除（TTL），不需要逐条清理
- 查询时按天查找，大部分场景只需要查最近 1-3 天

**建议**：key 设计和分片策略是 Redis 面试的重点，准备好"为什么这么分"的解释。

---

### Q12: "异步更新 + 批量同步，数据库 TPS 降至 1/50"——这个怎么实现的？

**答案**：
问题：每次用户标记"已读"都直接写 MySQL，高峰期 TPS 可达 5000+，数据库扛不住。

解决方案：
1. **异步更新**：前端标记已读 → 写入 Redis Bitmap（毫秒级）→ 立即返回成功
2. **批量同步**：定时任务（如每分钟）扫描 Redis 中的脏数据 → 批量组装成 `INSERT ... ON DUPLICATE KEY UPDATE` 的批量 SQL → 一次写入数据库
3. **效果**：
   - 原来：5000 次/秒 × 单条 INSERT = 5000 TPS
   - 优化后：每分钟 1 次批量写入，每次合并 ~3 万条 = ~100 TPS（60 秒平均）
   - TPS 降至约 1/50

**建议**：面试官可能追问"如果批量同步失败怎么办"——回答：Redis 数据还在，下次同步会重试；最坏情况 Redis 崩溃，已读状态丢失但不影响核心业务。

## 五、RabbitMQ 数据同步（了解性）

### Q13: 为什么用 RabbitMQ 异步通知 ES 更新索引，而不是同步双写？

**答案**：
- **同步双写**：MySQL 写入成功后同步更新 ES，如果 ES 更新失败会导致 MySQL 事务回滚或数据不一致
- **MQ 异步**：MySQL 写入成功后发消息到 RabbitMQ → 消费者异步更新 ES
  - 好处 1：**解耦**——MySQL 写入不依赖 ES 的可用性
  - 好处 2：**削峰**——批量下架 1000 个商品时不会瞬间打满 ES
  - 好处 3：**可靠性**——MQ 消息持久化 + 消费失败重试，保证最终一致性

**建议**：经典的"数据同步"面试题，重点说清楚"为什么不同步"。

---

### Q14: MQ 消息丢失怎么办？如何保证 ES 和 MySQL 的最终一致性？

**答案**：
三段保障：
1. **生产端可靠投递**：开启 RabbitMQ 的 Publisher Confirm 机制，确认消息已到达 Broker
2. **Broker 持久化**：消息 + 队列都设为 durable，避免 Broker 重启丢消息
3. **消费端可靠消费**：手动 ACK 模式，消费成功后才 ACK；消费失败进入死信队列，人工排查

补偿机制：定时全量对比任务（如凌晨跑一次），扫描 MySQL 和 ES 的数据差异，发现不一致则修复。

**建议**：MQ 可靠性保障是面试必问题，"生产端 + Broker + 消费端"三段式回答最标准。

---

### Q15: 消息重复消费怎么处理？

**答案**：
ES 的 index 操作天然幂等——相同 `_id` 的文档多次 index 只会覆盖不会重复。所以即使消息重复消费，最终结果也是正确的。

但如果涉及非幂等操作（如计数器递增），需要额外处理：
- 消息带唯一 ID（如 UUID）
- 消费前先查 Redis 是否已处理过（幂等表）
- 已处理则跳过，未处理则执行 + 标记

**建议**：先说"当前场景天然幂等"，再说"通用方案"，体现你既了解业务特点又懂通用解法。

## 六、"降低人工客服 90% 基础工作量"（追问题）

### Q16: 90% 这个数字怎么来的？

**答案**：
基于客服工单的分类统计：
- 基础工作（商品咨询、订单查询、物流跟踪、退换货政策）占全部工单的约 90%，这些都可以被 AI 自动处理
- 复杂工作（投诉、纠纷、特殊退款审批等）占约 10%，仍需人工
- "降低 90% 基础工作量"不等于"裁掉 90% 客服"，而是让人工客服从重复劳动中解放出来，专注处理复杂问题

**建议**：精确描述"降低基础工作量"和"降低人力"的区别，避免面试官挑刺。

---

### Q17: 你觉得这个客服系统目前最大的不足是什么？

**答案**：
1. **多轮对话能力弱**：当前主要处理单轮问答，对需要多轮澄清的复杂问题处理不好（如"我买的那个，就是上次那个红色的，能退吗"）
2. **情感识别缺失**：无法检测用户情绪，对于生气/焦虑的用户仍然机械式回答
3. **知识更新延迟**：商品/政策变更后，RAG 索引的更新有延迟（MQ 异步的固有特性）
4. **缺少反馈闭环**：用户对 AI 回答不满意时没有有效的反馈收集机制

**建议**：自我反思是面试加分项，表明你在持续思考而非"做完就算"。
