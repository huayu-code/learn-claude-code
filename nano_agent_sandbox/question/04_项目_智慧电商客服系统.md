## 一、项目背景 & 整体架构（了解性）

### Q1: 这个项目解决什么业务问题？为什么要做智能客服？

**答案**：
传统电商客服的痛点：
1. **人工成本高**：需要大量客服人员 7×24 小时在线
2. **响应慢**：高峰期用户排队等待时间长
3. **质量不稳定**：不同客服回答质量参差不齐
4. **重复劳动**：80%+ 的问题是重复的（物流查询、退换货政策等）

我的系统目标：
- 通过 **RAG + Tool Calling** 实现 AI 自动应答
- 满足 **95%** 的客服查询需求
- 降低人工客服 **90%** 的基础工作量
- 初步具备 **客服 Agent** 能力——AI 不仅能回答，还能执行操作

**建议**：先讲业务痛点，再讲技术方案，面试官更容易理解你在解决什么问题。

## 二、RAG 系统（核心 — 深度题）

### Q3: RAG 是什么？你的 RAG 系统怎么实现的？

**答案**：
**RAG（Retrieval Augmented Generation）**：检索增强生成。核心思想是让 LLM 回答问题时，先从知识库检索相关文档，再结合检索结果生成回答。

我的实现流程：
```
用户问题："怎么退货？"
    ↓
1. Embedding：将问题向量化（混元/OpenAI Embedding）
    ↓
2. 检索：ES 向量相似度搜索，返回 Top 5 相关文档
   - 退换货政策.md（相似度 0.92）
   - 七天无理由退货说明.md（相似度 0.87）
   - ...
    ↓
3. 重排序（可选）：根据相关性重排
    ↓
4. 构建 Prompt：
   "根据以下知识回答用户问题：
    [知识1] 退换货政策：...
    [知识2] 七天无理由：...
    用户问题：怎么退货？"
    ↓
5. LLM 生成：基于检索到的知识生成回答
```

**建议**：RAG 是 AI 应用的核心技术，一定要能画出完整流程。

### Q4: Elasticsearch 怎么做向量检索？和传统全文检索有什么区别？

**答案**：
ES 8.x 版本支持 **dense_vector** 字段类型：

```json
// 索引映射
{
    "mappings": {
        "properties": {
            "content": {"type": "text"},
            "embedding": {
                "type": "dense_vector",
                "dims": 1024,  // 向量维度
                "index": true,
                "similarity": "cosine"  // 余弦相似度
            }
        }
    }
}
```

向量检索 vs 全文检索：

| 维度 | 全文检索 (BM25) | 向量检索 |
|------|----------------|---------|
| **原理** | 关键词匹配 + TF-IDF | 语义相似度 |
| **查询** | "退货政策" | [0.1, 0.2, ..., 0.3]（1024维） |
| **优势** | 精确匹配、可解释 | 理解语义、同义词 |
| **劣势** | 无法理解语义 | 需要 Embedding 模型 |

实际使用：**混合检索**（BM25 + 向量）效果最好
```json
{
    "query": {
        "bool": {
            "should": [
                {"match": {"content": "退货"}},  // BM25
                {"knn": {"embedding": {...}}}    // 向量
            ]
        }
    }
}
```

**建议**：能说出"混合检索"比单一方式效果好，体现你的实践经验。

### Q5: 知识库包含哪些内容？怎么构建和更新？

**答案**：
知识库覆盖三大类：
1. **商品咨询**：商品详情、规格参数、使用说明
2. **退换货政策**：七天无理由、质量问题退换、流程说明
3. **活动规则**：满减规则、优惠券使用、限时活动

构建流程：
```
原始文档（Word/Markdown/PDF）
    ↓
文档解析：提取纯文本
    ↓
分块（Chunking）：每 500 字一块，重叠 100 字
    ↓
Embedding：调用混元/OpenAI 接口向量化
    ↓
入库 ES：content + embedding + metadata
```

更新机制：
- **全量更新**：运营修改文档后，触发重新构建
- **增量更新**：RabbitMQ 监听文档变更事件，只更新变化的部分

**建议**：分块策略（chunk size、overlap）是 RAG 调优的关键，可以展开讲。

### Q6: 分块策略为什么是 500 字一块、重叠 100 字？

**答案**：
**500 字**的选择：
- 太小（如 100 字）：上下文不完整，检索到的片段缺乏语义
- 太大（如 2000 字）：噪音太多，不相关内容干扰回答
- 500 字：约等于一个完整段落，语义完整且信息密度高

**重叠 100 字**的原因：
- 防止关键信息被切断在边界
- 示例：一句话 "退货需要在收货后 7 天内申请" 如果刚好被切在 "7 天" 处，没有重叠就会丢失完整信息

**实践调优**：
- 根据实际问答效果调整
- 可以按文档类型设置不同策略（政策类用 300 字，商品类用 500 字）

**建议**：这些数字不是拍脑袋定的，要能解释 trade-off。

## 三、Tool Calling（核心 — 深度题）

### Q7: Tool Calling 是什么？怎么让 AI 能执行具体操作？

**答案**：
**Tool Calling（Function Calling）**：让 LLM 不仅能回答问题，还能调用外部工具/API。

原理：
1. 在 Prompt 中告诉 LLM 有哪些可用工具（JSON Schema 格式）
2. LLM 判断需要调用工具时，返回结构化的工具调用请求
3. 系统执行工具，将结果返回给 LLM
4. LLM 基于工具结果生成最终回答

示例：
```
用户："我的订单 12345 到哪了？"
    ↓
LLM 返回：{"tool": "query_logistics", "args": {"order_id": "12345"}}
    ↓
系统调用物流接口 → 返回 "已到达杭州转运中心"
    ↓
LLM 生成："您的订单 12345 已到达杭州转运中心，预计明天送达。"
```

**建议**：Tool Calling 是 Agent 的核心能力，一定要能画出完整流程。

## 四、Redis Bitmap（深度题）

### Q10: 为什么用 Redis Bitmap 存储已读状态？节约 98% 空间怎么计算的？

**答案**：
**场景**：用户打开聊天窗口时，需要标记消息为"已读"。每条消息对应一个已读状态。

**传统方案**：
```sql
CREATE TABLE message_read_status (
    message_id BIGINT,
    user_id BIGINT,
    is_read BOOLEAN,
    PRIMARY KEY (message_id, user_id)
);
```
每条记录约 20 字节（8+8+1+索引开销）

**Bitmap 方案**：
```
Key: read_status:{user_id}
Value: Bitmap（每个 bit 代表一条消息是否已读）

SETBIT read_status:12345 1001 1  // 消息 1001 已读
GETBIT read_status:12345 1001    // 查询是否已读
```

空间对比（假设 100 万用户，每用户 1000 条消息）：
- **传统方案**：100万 × 1000 × 20 字节 = 20 GB
- **Bitmap 方案**：100万 × (1000 / 8) 字节 = 125 MB
- **节约**：(20GB - 125MB) / 20GB ≈ **99.4%**

取 98% 是保守估计（考虑 Redis 的额外开销）。

**建议**：Bitmap 是 Redis 的经典应用场景，一定要能手算空间节约比。

---

### Q11: Bitmap 有什么局限性？什么场景不适合？

**答案**：
局限性：
1. **消息 ID 必须是连续整数**：Bitmap 的 offset 对应消息 ID，如果 ID 是 UUID 或稀疏的，会浪费大量空间
2. **只能存布尔值**：只能表示"已读/未读"，无法存储"已读时间"等附加信息
3. **单个 Key 有大小限制**：Redis 的 String 最大 512MB = 40 亿 bit，超大用户需要分片

解决方案：
```
// 分片策略：每 100 万条消息一个 Bitmap
Key: read_status:{user_id}:{message_id / 1000000}
Offset: message_id % 1000000
```

不适合的场景：
- 消息 ID 是 UUID
- 需要存储已读时间
- 消息总量超过 40 亿

**建议**：说出局限性和解决方案，体现你对技术边界的理解。

## 五、异步更新 + 批量同步（深度题）

### Q12: "已读状态写回数据库"为什么要异步？"批量同步"怎么实现？

**答案**：
**为什么异步**：
- 已读状态是高频操作：用户每打开一条消息就触发一次
- 同步写数据库：每次请求都要等 DB 响应，RT 增加 5-10ms
- 异步写：请求立即返回，后台批量写入

**批量同步实现**：
```java
@Component
public class ReadStatusSyncService {
    
    private final BlockingQueue<ReadStatus> queue = new LinkedBlockingQueue<>(10000);
    
    // 接收已读事件
    public void markAsRead(long messageId, long userId) {
        // 1. 立即更新 Redis Bitmap（用户可见）
        redisTemplate.opsForValue().setBit(key, messageId, true);
        // 2. 放入队列（异步写 DB）
        queue.offer(new ReadStatus(messageId, userId));
    }
    
    // 定时批量写入
    @Scheduled(fixedRate = 1000)  // 每秒执行一次
    public void batchSync() {
        List<ReadStatus> batch = new ArrayList<>();
        queue.drainTo(batch, 500);  // 最多取 500 条
        if (!batch.isEmpty()) {
            // 批量 INSERT
            readStatusMapper.batchInsert(batch);
        }
    }
}
```

**效果**：
- 原方案：1000 QPS → 1000 次 DB 写入
- 新方案：1000 QPS → 每秒 2 次批量写入（每次 500 条）
- DB TPS 降至原来的 **1/500 ≈ 1/50**

**建议**：异步 + 批量是高并发场景的标配方案，要能手写代码。

---

### Q13: 异步队列满了怎么办？数据会丢失吗？

**答案**：
**队列满的情况**：
- `queue.offer()` 返回 false，表示队列满
- 需要降级策略

**防丢失方案**：
```java
public void markAsRead(long messageId, long userId) {
    redisTemplate.opsForValue().setBit(key, messageId, true);  // Redis 先写
    
    boolean offered = queue.offer(new ReadStatus(messageId, userId));
    if (!offered) {
        // 降级：直接同步写 DB
        readStatusMapper.insert(new ReadStatus(messageId, userId));
        log.warn("Queue full, sync write to DB");
    }
}
```

**数据一致性**：
- **Redis 是主**：用户看到的已读状态以 Redis 为准
- **MySQL 是备**：用于持久化和数据分析
- 即使 MySQL 写入延迟，用户体验不受影响

**建议**：异步方案一定要考虑"队列满"和"数据丢失"的边界情况。

## 六、RabbitMQ 异步通知（深度题）

### Q14: 为什么用 RabbitMQ 同步 MySQL 和 ES？为什么不用定时任务轮询？

**答案**：
**场景**：商品下架、库存变化等信息需要同步到 ES，保持检索结果的准确性。

**定时任务轮询的问题**：
- 轮询间隔太长：数据不一致窗口大（用户搜到已下架商品）
- 轮询间隔太短：DB 压力大（频繁全表扫描）
- 无法感知"哪条数据变了"：每次都要全量比对

**MQ 异步通知的优势**：
- **实时性**：数据变更立即发消息，ES 秒级更新
- **解耦**：业务代码只管发消息，ES 更新逻辑独立
- **削峰**：大促期间大量更新，MQ 缓冲防止 ES 过载

```java
// 商品服务：下架商品时发消息
public void offShelf(Long productId) {
    productMapper.updateStatus(productId, "OFFLINE");
    rabbitTemplate.convertAndSend("product.exchange", "product.offline", productId);
}

// ES 服务：消费消息更新索引
@RabbitListener(queues = "product.offline.queue")
public void onProductOffline(Long productId) {
    esClient.delete("products", productId);
}
```

**建议**：MQ 解耦是微服务架构的基本功，要能说清楚为什么不用定时任务。

---

### Q15: MQ 消息丢失怎么办？消息重复消费怎么处理？

**答案**：
**消息丢失的三个环节**：
1. **生产者→MQ**：开启 Publisher Confirm
2. **MQ 存储**：开启持久化（durable queue + persistent message）
3. **MQ→消费者**：手动 ACK

```java
// 生产者确认
rabbitTemplate.setConfirmCallback((data, ack, cause) -> {
    if (!ack) {
        log.error("Message send failed: {}", cause);
        // 重试或记录失败
    }
});

// 消费者手动 ACK
@RabbitListener(queues = "product.offline.queue", ackMode = "MANUAL")
public void onProductOffline(Long productId, Channel channel, Message message) {
    try {
        esClient.delete("products", productId);
        channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
    } catch (Exception e) {
        // 处理失败，消息重回队列
        channel.basicNack(message.getMessageProperties().getDeliveryTag(), false, true);
    }
}
```

**重复消费的处理（幂等性）**：
```java
public void onProductOffline(Long productId) {
    // 幂等检查：ES 中已经不存在，说明已处理过
    if (!esClient.exists("products", productId)) {
        return;  // 跳过重复消息
    }
    esClient.delete("products", productId);
}
```

**建议**：MQ 的可靠性保障是高频面试题，要能说出"三个环节 + 幂等处理"。

## 七、客服 Agent 能力（追问题）

### Q16: 你说"初步具备客服 Agent 能力"，和传统客服系统有什么区别？

**答案**：
| 维度 | 传统客服系统 | 客服 Agent |
|------|-------------|-----------|
| **回答方式** | 关键词匹配 → 固定话术 | LLM 理解语义 → 生成回答 |
| **操作能力** | 无，只能回答 | Tool Calling 执行业务操作 |
| **上下文** | 无记忆 | 记住对话历史 |

**"初步"的含义**：
- 能处理 95% 的常见问题
- 但复杂场景（投诉、特殊退款）仍需人工
- 没有做完整的 Agent Loop（如自动重试、错误恢复）

**建议**：诚实说明"初步"的边界，不要过度吹嘘。

## 八、设计权衡 & 扩展（追问题）

### Q18: 如果 QPS 从 1000 增长到 10 万，系统需要怎么改造？

**答案**：
瓶颈分析：
1. **LLM 调用**：延迟高、成本高 → 加缓存、用更快的模型
2. **ES 检索**：可以水平扩展 → 加副本
3. **MySQL**：写入瓶颈 → 读写分离、分库分表
4. **Redis**：单实例瓶颈 → 集群模式

改造方案：
```
用户请求 → 网关（限流 + 熔断）
    ↓
负载均衡 → 多个 Spring Boot 实例
    ↓
Redis Cluster（读缓存）→ 命中则返回
    ↓
ES Cluster（向量检索）→ 多副本分担读压力
    ↓
LLM 调用（加缓存）→ 相同问题命中缓存
    ↓
MySQL（读写分离）→ 主写从读
```

**建议**：这是经典的"扩展性"面试题，要能从瓶颈分析到解决方案。

### Q19: 这个项目最大的技术挑战是什么？

**答案**：
最大挑战是 **RAG 检索的准确率优化**。

问题：
- 初版检索准确率只有 70%——用户问"怎么退货"，检索到的是"退款到账时间"
- 原因：embedding 模型把"退货"和"退款"判断为相似

解决方案：
1. **混合检索**：BM25（关键词）+ 向量（语义）融合
2. **查询改写**：用 LLM 将用户问题扩展为多个查询
3. **重排序**：用交叉编码器（Cross-Encoder）对 Top 10 结果重排
4. **知识库结构化**：按主题分类（退货 / 退款 / 物流），先分类再检索

效果：准确率从 70% 提升到 92%。

**建议**：RAG 优化是实际项目中的核心难点，能讲出具体优化方案很加分。

### Q20: 和你的 Self-Evolving Code Agent 项目对比，两者的 Agent 能力有什么异同？

**答案**：
**相同点**：
- 都用了 **Tool Calling** 让 AI 执行具体操作
- 都有 **上下文管理**（对话历史）
- 都是 **LLM 驱动**的智能系统

**不同点**：
| 维度 | 智慧电商客服 | Self-Evolving Code Agent |
|------|-------------|-------------------------|
| **工具来源** | 人工预定义（订单查询、物流跟踪） | **LLM 自动创建** |
| **决策复杂度** | 单轮问答为主 | **ReAct 多轮循环** |
| **安全机制** | 业务接口有权限控制 | **AST + Sandbox + Harness 三层** |
| **核心创新** | RAG + 业务集成 | **工具自进化** |

**建议**：能把两个项目关联起来对比，体现你对 Agent 技术的系统性理解。
